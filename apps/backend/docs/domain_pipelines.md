# domain/pipelines

> The run engine: data-defined pipelines, run + stage lifecycle, HITL pauses.

## Purpose

Owns the four tables backing the pipelines run engine (`pipelines`, `pipeline_runs`, `stage_executions`, `run_pauses`), the definition model + validation, the CRUD surface behind `/api/pipelines`, and the run engine itself (`engine.py`) — the `ROUTE_RUN`/`START_STAGE`/`HANDLE_AGENT_EVENT` taskiq trio that drives a `PipelineRun` from `queued` through `action`/`skill`/`review` stage execution to a terminal state. This module replaces `core/workflow` + `domain/reviewer`'s workflow-engine role; both stay alive and untouched during coexistence (`core/workflow`, `domain/reviewer`, `pr_review_v1`) — `handle_agent_event` and `core/workflow.handle_agent_event` are both registered as `core/agent_gateway` consumers (see [core_agent_gateway.md § Agent-event consumer registry](core_agent_gateway.md#agent-event-consumer-registry)) and each ignores terminal events it doesn't own. Definition CRUD, `start_run`, `request_cancel` (incl. immediate cancel of a `paused` run), skill-stage dispatch (with engine-managed workspace provision/cleanup), the review-fix loop with durable findings (`domain/findings`), real boundary evaluation (`boundary.py`) after every skill/review stage, and `resolve_pause`'s `approve`/`kill` actions are real. `resolve_pause`'s `instruct`/`send_back` actions raise a labeled `NotImplementedError` — the re-entry machinery that resumes a stage with an instruction, or rewinds to an earlier stage, doesn't exist yet. `start_rerun_from_stage`, `instantiate_template`, `list_templates`, `list_runs_for_ticket`, and `get_run_overview` are still stubs raising `NotImplementedError`; `has_run_in_flight` is real.

## Public interface

Definition-model value objects (`PipelineDefinition`, `Stage` discriminated union — `SkillStage | ReviewSkillStage | ActionStage | PipelineCallStage`, `BoundaryControl`, `ReviewConfig`), stored-entity VOs (`Pipeline`, `PipelineSummary`), run-lifecycle value objects (`Kickoff`, `PipelineRun`, `StageExecution`, `RunOverview`, `PauseResolution`), the function surface (`create_pipeline`, `update_pipeline`, `delete_pipeline`, `get_pipeline`, `list_pipelines`, `pipeline_referenced_by_call`, `start_run`, `request_cancel`, `resolve_pause`, `has_run_in_flight` — real (`resolve_pause`'s `instruct`/`send_back` actions raise `NotImplementedError`); `start_rerun_from_stage`, `instantiate_template`, `list_templates`, `list_runs_for_ticket`, `get_run_overview` — stub), and the typed error hierarchy (`PipelineNotFoundError`, `PipelineNameTakenError`, `PipelineValidationError`, `PipelineReferencedError`, `RunNotFoundError`, `RunAlreadyTerminalError`, `PauseNotFoundError`, `PauseAlreadyResolvedError`, `NotEscalationTargetError`, `StageNotInDefinitionError`, `MissingInheritedArtifactError`).

`flatten()`, `validate_definition()`, and `FlattenedDefinition.from_snapshot()` (in `definition.py`) are internal to the module — not re-exported. `engine.py` (the `ROUTE_RUN`/`START_STAGE`/`HANDLE_AGENT_EVENT` task bodies, `attempt_promotion`, `promote_oldest_queued`, `cancel_queued`, `resume_from_pause`, `kill_run`, `cancel_paused`, `get_open_pause_for_run`), `boundary.py` (`evaluate_boundary`, `BoundaryDecision`), and `escalation.py` (`resolve_escalation_targets`, `is_pause_responder`) are likewise intra-module only — `service.py`'s `start_run`/`request_cancel`/`resolve_pause` are the sole public entry points onto the engine. `contracts.py` (`SkillReturn`, `SkillReviewReturn`, `SkillReviewFinding`, `PriorFindingVerdict`, `bucket_confidence`, confidence-cutoff constants) is intra-module — both skill contracts ride `Invocation.context["output_schema"]`, no external caller needs them.

HTTP routes: `GET/POST /api/pipelines`, `GET/PUT/DELETE /api/pipelines/{id}`, `POST /api/pipelines/runs/{run_id}/cancel`, `POST /api/pipelines/runs/pauses/{pause_id}/respond` — see § HTTP endpoints below. Further run-lifecycle routes (rerun, run/overview reads) land with the rest of the run engine.

## Module architecture

### Entities

- **Pipeline** — the stored org pipeline: a `PipelineDefinition` plus persistence metadata (`updated_at`, `updated_by_login`, `referenced`).
- **PipelineRun** — one run of a pipeline against a ticket; replaces `WorkflowExecution`. Pins a flattened definition snapshot at start.
- **StageExecution** — one execution attempt of one stage (or a `kind='system'` bookkeeping row — provision/cleanup/auth-refresh/push-branch).
- **RunPause** — one HITL pause; replaces `PendingHumanDecision`.

### Key value objects

- **PipelineDefinition** — the authored content (`id`, `name`, `description`, `stages`); a discriminated union on `kind` (`skill | review | action | call`). `id` (top-level and per-stage) defaults to a fresh uuid7 at parse time (`Field(default_factory=uuid7)`) — a request body that omits `id` on a new pipeline or a newly-added stage gets one server-minted for free.
- **Kickoff** — intake point + actor + input text that started a run; the ticket (with title) exists before the run.
- **BoundaryControl** — flat per-stage "what to do next" setting (`always_hitl | always_proceed | conditional`).
- **RunOverview** — server-computed Overview-tab payload, tagged `paused | in_flight | terminal`.

### Core user flows

- **Create/update a pipeline** — `create_pipeline`/`update_pipeline` dry-run `validate_definition()`: flattens the edited definition AND every org pipeline that transitively calls it (via `PipelineCallStage`), against a supplied org-wide id → definition map. A call cycle, an unresolvable call target, or a duplicate flattened stage name (skill/review stages only — action/call stages carry no name) raises `PipelineValidationError` before anything is written. Storage never flattens — `stages` keeps `PipelineCallStage` entries as-is, so a callee's edits reach every caller's *future* runs; the real per-run flatten is the run engine's job.
- **Delete a pipeline** — refused with `PipelineReferencedError` when `pipeline_referenced_by_call` (an app-side scan of the org's `stages` JSONB for a `call` stage targeting this pipeline) or `repos.pipeline_referenced_by_binding` (stub — always `False` until repo trigger bindings exist) is true.
- **Read** — `get_pipeline`/`list_pipelines` resolve `updated_by_login` via `core/tenancy.get_membership_info` (the org membership handle, not a global username) and `referenced` via the same OR check delete uses.
- **Start a run** — `start_run` flattens the pipeline against *current* org definitions and pins the result as `pipeline_runs.definition_snapshot`. The row always inserts `state='queued'` first; `engine.attempt_promotion` then attempts a conditional `queued -> running` flip guarded by `ux_pipeline_runs_one_in_flight` (SAVEPOINT-wrapped — an `IntegrityError` from a concurrent promotion on the same ticket leaves the run `queued`, never raises). A successful promotion stamps `tickets.current_run_id`, flips the ticket `pending -> running` (`transition_ticket_on_run_start`), writes `run.started`, publishes `run_state_changed`, and enqueues the bootstrap `ROUTE_RUN`.
- **Run a stage** — `ROUTE_RUN` decides the next boundary action (dispatch stage 0, dispatch the next stage, or wind down to a terminal state); `START_STAGE` executes one stage and reports back to `ROUTE_RUN` (directly for `action`; via `HANDLE_AGENT_EVENT` for `skill`/`review`, once the agent's terminal event arrives). `kind='action'` stages run synchronously: `Action.execute` runs inside a SAVEPOINT (the `stage_executions` row's `status`/`action_result`/`failure_reason` land atomically with the action's own writes). `kind='skill'`/`kind='review'` stages provision a workspace first when the run doesn't have one yet (`run.workspace_id IS NULL` → a `kind='system'` `provision-workspace` stage execution dispatches and parks; on success the real stage dispatch resumes at `run.current_stage_index`), then dispatch the invocation for real (see below). An action-only pipeline never provisions, so it never runs cleanup either — the permanent behavior for zero-skill pipelines, not a temporary gap.
- **Dispatch a skill stage** — `_dispatch_skill_stage` mints `command_id` first (needed for the skill's `artifact_path = "$TMPDIR/<command_id>.md"`), resolves `StageInvocationContext.input` (first stage: the kickoff's `input_text`; else the nearest upstream `SkillStage`'s latest final artifact body, walking backward, skipping `ActionStage`/`ReviewSkillStage` — neither produces one; falls back to the kickoff input if none found), builds the `Invocation` (with `SkillReturn.model_json_schema()` injected as `context["output_schema"]`), and calls `core/coding_agent.dispatch_invocation` — parking on `pending_agent_command_id`. `HANDLE_AGENT_EVENT` resumes on the terminal event: infra failures (non-`success` `outcome_label`) fail the stage with the reported reason; a `SkillReturn` schema violation or a non-`completed` outcome (`send_back`/`cannot_complete` — run-failure placeholders until the boundary/send-back re-entry machinery exists) fails the stage; a `completed` outcome without an artifact body fails the stage (artifact required iff `outcome == "completed"`); otherwise the artifact is stored via `domain/artifacts.store` and immediately marked final via `mark_final` (nothing reads a mid-loop artifact downstream, so marking every produced version final is harmless independent of the boundary outcome), the artifact id + raw confidence + `paths_affected` land in `loop_state`, `stage_executions.confidence` is bucketed (`bucket_confidence`) from the last **main**-skill `SkillReturn` (a review pass's confidence lands only in `loop_state`). When `stage.review` is `None` this calls `_settle_stage_boundary` (below) directly; when configured, the review-fix loop takes over first and calls it once the loop stops.
- **Review-fix loop** — a `SkillStage` with `review` configured runs `main → (review → fix)* → boundary` inside the *same* `stage_executions` row (`phase` cycles `main → review → fix → review → …`; `iteration` counts review passes, bumped by `_dispatch_review_invocation` before each one). The review dispatch speaks `SkillReviewReturn` over the artifact just produced, with `prior_findings` = this stage execution's own findings (any status) ∪ the ticket's other open durable findings (`_resolve_prior_findings`). On return, every `SkillReviewFinding` materializes via `findings.record_findings` (engine-minted uuid7 ids), then the verdict matrix applies mechanically for each `PriorFindingVerdict` (`_apply_verdicts`): `fixed → findings.resolve`; `still_present → findings.reflag` when the referenced finding is open, `findings.reopen` when resolved; `user_overrode → findings.dismiss`. Residuals = this stage execution's still-`open` findings, recorded on the `review`-phase `loop_state` entry every pass. While residuals remain and `iteration < review.max_iterations`, `_dispatch_fix_invocation` reruns the main skill with `revision=RevisionContext(source="fix", text=<rendered residuals>, prior_artifact=<last artifact body>)` — the rendered text + `source` are also stamped onto `stage_executions.revision` (independent of the wire payload, so the fix's input is durably observable) — then re-reviews the new artifact. The loop stops once residuals are empty or the iteration cap is hit, then `_settle_stage_boundary` runs. A standalone `ReviewSkillStage` (`kind='review'`) dispatches exactly one `SkillReviewReturn` invocation the same way (findings, verdicts, residuals) but stores no artifact and never loops, then calls `_settle_stage_boundary` with `paths_affected=()` (`SkillReviewReturn` carries no such field, so `on_protected_code` can never trip for it) — `_dispatch_review_only_stage` / `_handle_review_stage_event`.
- **Evaluate a stage boundary** — `_settle_stage_boundary` computes residuals (this stage execution's still-`open` findings — empty for a review-off stage), fetches the ticket's `repo_external_id`, and calls `boundary.evaluate_boundary(stage.boundary, residuals=…, paths_affected=…, confidence=stage_executions.confidence, …)`. `cannot_complete`/`send_back` main-skill outcomes never reach it — they fail the stage earlier (see above), so the stage's own work has always "completed" by this point. `always_proceed`/`always_hitl` short-circuit; `conditional` accumulates every tripped condition (`on_blocker_residuals`/`on_should_fix_residuals` against residual severities, `on_protected_code` against `repos.evaluate_protected(paths_affected)`, `on_confidence_below` against the bucketed confidence) rather than stopping at the first hit. `proceed` marks the stage `completed`/`boundary_outcome="proceeded"` and enqueues `ROUTE_RUN`; `pause` calls `_enter_pause`.
- **Enter a pause** — `_enter_pause` marks the stage `boundary_outcome="paused"` (`boundary_detail` = the tripped conditions), resolves the escalation set (`escalation.resolve_escalation_targets` unioned with `BoundaryDecision.protected_owner_user_ids` when `on_protected_code` fired), inserts an open `run_pauses` row, flips `pipeline_runs.state='paused'`, extends the run's workspace expiry by `PAUSED_RUN_WORKSPACE_GRACE_SECONDS` (`core/workspace.extend_expiry`) when one exists, flips the ticket to `hitl` (`transition_ticket_on_run_paused`), publishes `run_state_changed`, writes `run.paused`, and fans out a `pipeline_run_paused` notification (`core/notifications.create`) to every resolved escalation target.
- **Resolve a pause** — `resolve_pause` (in `service.py`) loads the `run_pauses` row (org-scoped), 403s via `NotEscalationTargetError` unless the actor is in `escalation_user_ids` or an org admin/owner (`escalation.is_pause_responder`), 409s via `PauseAlreadyResolvedError` if already resolved. `approve` calls `engine.resume_from_pause`: flips the run back to `running`, the ticket back to `running` (`transition_ticket_on_run_resumed`), and enqueues `ROUTE_RUN(completed_stage_index=<the paused stage>, outcome_label="success")` — the exact call the stage would have made itself had the boundary not tripped. `kill` calls `engine.kill_run`, which routes through the normal terminal path (`_finish_or_cleanup(… "killed" …)`) — same cleanup-first routing as any other decided outcome. `instruct`/`send_back` raise a labeled `NotImplementedError`: the re-entry machinery that resumes a stage with an instruction, or rewinds to an earlier stage, doesn't exist yet. Every resolution (including the `NotImplementedError` ones raised before this point) writes `run.pause_resolved` first for `approve`/`kill`.
- **Reach a terminal state** — once `ROUTE_RUN` decides a target state (`completed`/`failed`/`cancelled`/`killed`), `_finish_or_cleanup` dispatches the `cleanup-workspace` system stage first when the run ever provisioned a workspace (`run.workspace_id IS NOT NULL`); `_finalize_after_cleanup` re-derives the same target state once cleanup's terminal event arrives (cleanup's own failure is logged, not re-surfaced — the run's outcome was already decided) and calls `_enter_terminal`. A run that never provisioned (action-only) enters the terminal state directly. `_enter_terminal` writes the terminal `pipeline_runs.state`, flips the ticket to the matching status (`transition_ticket_on_run_terminal`), writes `run.{state}`, publishes `run_state_changed`, notifies the resolved escalation targets on `failed` (`escalation.resolve_escalation_targets` — kickoff actor, else schedule notify list, else the PR author's linked identity, else the org's admins), and promotes the oldest `queued` run on the ticket (`promote_oldest_queued`).
- **Cancel a run** — `request_cancel`: `queued` cancels immediately (`engine.cancel_queued`); `running` sets `cancel_requested` and the next `ROUTE_RUN` boundary check routes to `cancelled` instead of the next stage — checked on every boundary including the last one, so a run cannot slip into `completed` after cancellation was requested; `paused` cancels immediately too (`engine.cancel_paused` — closes the open pause row with `resolved_at` stamped and no `resolution` recorded, then routes through the same cleanup-first terminal path). Raises `RunAlreadyTerminalError` on an already-terminal run (409 at HTTP).

### State machines

- **Run state**: `queued → running ⇄ paused → completed | failed | killed | cancelled`. `paused` is entered by `_enter_pause` when a stage's `evaluate_boundary` call decides `pause`; `resolve_pause(approve)` flips back to `running`, `resolve_pause(kill)` or `request_cancel` on a `paused` run flips to a terminal state.
- **Run phase**: `provision → stages → cleanup`. `'provision'` while a workspace provision (initial or re-provision) is in flight; `'stages'` for normal stage dispatch/execution; `'cleanup'` while the terminal-bound cleanup system stage is in flight. An action-only run never leaves `'stages'` — see § Core user flows.
- **Workspace liveness check.** Before every skill/review-stage dispatch, `_workspace_is_live` calls `core/workspace.get_workspace_info` and checks `status == ACTIVE`. A dead or absent workspace (any reason: reaper, idle timeout, agent restart, pause-grace elapsed) triggers `_dispatch_provision_stage`, which overwrites `run.workspace_id` with a new id and runs a fresh `provision-workspace` system row. Mid-run re-provision is structurally identical to the initial provision from the engine's perspective.
- **Auth-expired retry recovery.** An `auth_expired` failure on a skill stage's `main` phase inserts a `refresh-auth` system row, dispatches `core/workspace.dispatch_auth_refresh`, and parks. On refresh-auth success the skill invocation is retried once (new `stage_executions` row). The one-retry cap is tracked in `run.sendback_counts` under `auth_retry:{stage_name}` (same map as sendback tracking). A second `auth_expired` on the retry exhausts the cap and fails the run normally. An `auth_expired` failure during the `review`/`fix` phases (or on a standalone `review`-kind stage) is treated as a plain infra failure instead — the resume path only knows how to restart a stage fresh at `main`, and doing that mid-loop would silently discard `loop_state` and the iteration count.
- **`PAUSED_RUN_WORKSPACE_GRACE_SECONDS = 1800`** — module constant for the workspace-expiry extension `_enter_pause` grants via `core/workspace.extend_expiry`; the normal reaper collects the workspace once the deadline passes with no resolution.

### Deferred (not yet implemented)

- **`dispatch_push`** exists as a plain function (push-failure recovery after auth refresh) but has no caller yet — no stage currently pushes a branch from this engine.

## HTTP endpoints

`domain/pipelines/web.py`, prefix `/api/pipelines`, all `ORG_SCOPED`. Definition CRUD is gated by `Action.PIPELINES_MANAGE` (admin minimum); the run-cancel and pause-respond endpoints are gated by `Action.REVIEWER_WRITE` (builder minimum) — a run being watched on a ticket is a builder-facing concern, not a pipeline-authoring one. `respond`'s actual authorization is dynamic on top of that floor: the resolved escalation set union org admins/owners (`NotEscalationTargetError` → 403 otherwise).

| Method | Path | Response | Errors |
|---|---|---|---|
| GET | `/api/pipelines` | 200 `{pipelines: PipelineSummary[]}` | 403 |
| POST | `/api/pipelines` | 201 `{id}` | 400 `invalid_definition` · 409 `name_taken` |
| GET | `/api/pipelines/{id}` | 200 flat `{id, name, description, stages, updated_at, updated_by_login, referenced}` | 404 |
| PUT | `/api/pipelines/{id}` | 200 (same flat shape) | 400 `invalid_definition` · 404 · 409 `name_taken` |
| DELETE | `/api/pipelines/{id}` | 204 | 404 · 409 `referenced` |
| POST | `/api/pipelines/runs/{run_id}/cancel` | 202 | 404 · 409 `terminal` |
| POST | `/api/pipelines/runs/pauses/{pause_id}/respond` | 200 `{run_state}` | 400 `invalid_resolution` (`instruct`/`send_back`) · 403 `not_escalation_target` · 404 · 409 `already_resolved` |

POST/PUT bodies are the `PipelineDefinition` model itself — no separate write-request shape. `respond`'s body is `PauseResolution` (`action`, `instruction`, `send_back_to_stage`).

## Data owned

- `pipelines` — org-scoped pipeline definitions. `UNIQUE(org_id, name)`. `id` has no DB-level default (unlike its sibling tables below) — the pipeline's own id participates in `PipelineCallStage.pipeline_id` and templates ship with pinned ids, so `create_pipeline` mints it app-side via the definition model's `id` field (itself defaulting to a fresh uuid7 when the request omits it). See `apps/backend/docs/patterns.md` § UUID primary keys.
- `pipeline_runs` — one row per run. `UNIQUE INDEX ux_pipeline_runs_one_in_flight ON (ticket_id) WHERE state IN ('running','paused')` enforces one in-flight run per ticket at the DB layer; the run engine's promotion mechanics are built directly on this index.
- `stage_executions` — one row per stage-execution attempt. `CHECK` constraints on `kind`, `status`, `phase`, `confidence`, `boundary_outcome`. `action_result` is the `Action.Result` dump for `kind='action'` rows. `kind='system'` rows (`provision-workspace`, `cleanup-workspace`, `refresh-auth`) carry `stage_index=NULL`. `kind='skill'` rows carry `loop_state` entries per pass (`main`/`review`/`fix`, each with the produced artifact's id + raw confidence, or the review pass's new/residual finding ids) and `confidence` bucketed from the last main-skill pass onto the row; `revision` carries the last fix dispatch's `RevisionContext` dump (`source`, rendered `text`, `prior_artifact`). `kind='review'` rows (standalone `ReviewSkillStage`) carry one `review`-phase `loop_state` entry and no artifact.
- `run_pauses` — one row per HITL pause. `escalation_user_ids` is a `UUID[]`; `tripped` records which conditions fired (or `{"always_hitl": true}`); `resolution`/`instruction`/`send_back_to_stage`/`resolved_by`/`resolved_at` are null until `resolve_pause` applies. A cancelled-out-from-under pause has `resolved_at` set but `resolution` null.

## How it's tested

- `test/test_schema_service.py` (`@pytest.mark.service`) — the four owned tables accept a minimal insert; the one-in-flight partial unique index rejects a second concurrently-`running` run on the same ticket. Also seeds the five sibling-owned tables (`artifacts`, `pipeline_findings`, `repo_settings`, `repo_trigger_bindings`, `pr_comments`) via raw SQL to verify the full migrated schema end-to-end, since none of those modules' service functions exist yet to drive through their public API.
- `test/test_definition_flatten.py` (unit) — `flatten()`/`validate_definition()` over in-memory definition maps: nested call expansion, self + mutual cycles, unknown call targets, duplicate flattened stage names, and multi-hop transitive-caller revalidation on a callee edit.
- `test/test_pipeline_crud_service.py` (`@pytest.mark.service`) — full CRUD via `httpx.ASGITransport`: create + round-trip, cycle rejection, name-collision rejection, referenced-delete rejection, update, list, role gating (admin vs builder vs unauthenticated), and audit rows (`pipeline.created`, `pipeline.updated`, `pipeline.deleted`).
- `test/test_run_lifecycle_service.py` (`@pytest.mark.service`) — a two-action-stage run completes end-to-end through the `ROUTE_RUN`/`START_STAGE` trio: both `stage_executions` rows carry `action_result`, `run.started`/`run.completed` audit rows exist, `run_state_changed` publishes over SSE (via `test/drain.py`'s outbox-dispatch helper + `set_actions_for_tests`). A second test drives an `ActionError` to a `run.failed` terminal + a notification to the kickoff actor.
- `test/test_run_queueing_service.py` (`@pytest.mark.service`) — a second run started while the first is in flight sits `queued` until the first's terminal promotes it; three runs on one ticket (one `running`, two `queued`) prove the promotion-race guard directly (`attempt_promotion` on an already-occupied ticket returns `False`, never raises); cancel of a `running` run defers to the next boundary; cancel of a `queued` run is immediate.
- `test/test_skill_stage_service.py` (`@pytest.mark.service`) — one-skill-stage run: `provision-workspace` → skill stage (artifact stored+final) → `cleanup-workspace` → `run.completed`; missing artifact on `completed` outcome fails the stage; infra `completed_failure` fails the run; a dead workspace between stages triggers re-provision (two `provision-workspace` rows, run back in `provision` phase); `auth_expired` failure inserts a `refresh-auth` row, dispatches auth refresh, and on success retries the skill invocation; a second `auth_expired` exhausts the one-retry cap and fails the run normally. Uses `test/drain.py`.
- `test/test_event_consumer_seam_service.py` (`@pytest.mark.service`) — one `record_agent_event` call reaches both the old engine's and this engine's registered `HANDLE_AGENT_EVENT`; each no-ops on a `workflow_execution_id`/command it doesn't own; the old engine's flow (via `core/workflow.set_engine_for_tests`) still resumes to `DONE` through the shared registry.
- `test/test_review_loop_service.py` (`@pytest.mark.service`) — a one-skill-stage-with-review pipeline (`mode="always_proceed"` boundary — this file tests loop mechanics, not boundary policy) drives `main → review → fix → review`: iteration-1 findings materialize `open` with handles `<prefix>-001…`; the fix dispatch's `revision` (durably recorded on the row) contains the residual handles; a `fixed` verdict on iteration 2 resolves one finding with a `review_verdict` status event; the loop stops at `max_iterations` with the remaining residual recorded in `loop_state` even though it's still open. A second test drives a standalone `kind='review'` `ReviewSkillStage`: one invocation, findings materialize, zero `artifacts` rows.
- `test/test_boundary_pause_service.py` (`@pytest.mark.service`) — Acceptance: an `always_hitl` stage pauses the run with an open `run_pauses` row (`tripped={"always_hitl": true}`) and a `pipeline_run_paused` notification to the escalation user; `approve` by the escalation actor resumes to completion; a non-responder gets `NotEscalationTargetError`; `kill` terminates the run `killed` with the ticket transitioned; `request_cancel` on a `paused` run cancels immediately and closes the pause with no resolution. Conditional trips: `on_confidence_below` fires on a low-confidence main return; `on_protected_code` fires on a `paths_affected` path matching a `repos.put_settings`-configured protected set and folds the set's owner into the pause's escalation set.
