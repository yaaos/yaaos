# domain/pipelines

> The run engine: data-defined pipelines, run + stage lifecycle, HITL pauses, and shared stage-prompt renderer.

## Purpose

Owns the four tables backing the pipelines run engine (`pipelines`, `pipeline_runs`, `stage_executions`, `run_pauses`), the definition model + validation, the CRUD surface behind `/api/pipelines`, the shared stage-prompt renderer (`render_stage_prompt`), and the run engine itself (`engine.py`) — the `ROUTE_RUN`/`START_STAGE`/`HANDLE_AGENT_EVENT` taskiq trio that drives a `PipelineRun` from `queued` through `action`/`skill`/`review` stage execution to a terminal state. `handle_agent_event` is registered as a `core/agent_gateway` consumer (see [core_agent_gateway.md § Agent-event consumer registry](core_agent_gateway.md#agent-event-consumer-registry)) and ignores terminal events whose `run_id` isn't a `pipeline_runs` id. Definition CRUD, the run lifecycle (`start_run`/`request_cancel`, incl. immediate cancel of a `paused` run), skill-stage dispatch (with engine-managed workspace provision/cleanup), the review-fix loop with durable findings (`domain/findings`), real boundary evaluation (`boundary.py`) after every skill/review stage, every `resolve_pause` action (`approve`/`instruct`/`send_back`/`kill`), automatic send-back (a main skill's own `SkillReturn.send_back` or a residual finding's `defect_in_artifact`), `start_rerun_from_stage`, and the shipped-default template surface (`list_templates`/`instantiate_template`, backed by `defaults.py`) are all real. The Runs/Overview read models (`list_runs_for_ticket`, `get_run_overview`) live in `views.py`. `has_run_in_flight` is real. `scheduler_jobs.py` owns the two periodic `@scheduled` jobs — `pipeline_schedule_tick` (schedule-kind trigger bindings fire per minute) and `resume_stalled_runs` (the stall sweeper that makes Redis fully disposable for the engine) — see § Core user flows below.

## Public interface

Definition-model value objects (`PipelineDefinition`, `Stage` discriminated union — `SkillStage | ReviewSkillStage | ActionStage | PipelineCallStage`, `BoundaryControl`, `ReviewConfig`), stored-entity VOs (`Pipeline`, `PipelineSummary`), run-lifecycle value objects (`Kickoff`, `PipelineRun`, `StageExecution`, `RunOverview`, `PauseResolution`, `AttachmentRef`), the shared stage-prompt renderer (`render_stage_prompt(context, *, skill_directive, extra_directives=(), output_schema_mode="prompt") -> str` — used by `plugins/claude_code` to assemble the full stage invocation prompt; validates required context keys `{"stage_name", "input", "artifact_path"}`, raises `CodingAgentError` on missing), the function surface (`create_pipeline`, `update_pipeline`, `delete_pipeline`, `get_pipeline`, `list_pipelines`, `pipeline_referenced_by_call`, `start_run`, `start_manual_run`, `start_rerun`, `start_rerun_from_stage`, `request_cancel`, `resolve_pause`, `list_runs_for_ticket`, `get_run_overview`, `has_run_in_flight`, `register_run_terminal_hook`, `register_comment_findings_provider`, `list_templates`, `instantiate_template` — all real), and the typed error hierarchy (`PipelineNotFoundError`, `PipelineNameTakenError`, `PipelineValidationError`, `PipelineReferencedError`, `RunNotFoundError`, `RunNotRerunnableError`, `RunAlreadyTerminalError`, `PauseNotFoundError`, `PauseAlreadyResolvedError`, `NotEscalationTargetError`, `InvalidPauseResolutionError`, `StageNotInDefinitionError`, `MissingInheritedArtifactError`, `TemplateNotFoundError`, `RunInFlightError`).

`flatten()`, `validate_definition()`, and `FlattenedDefinition.from_snapshot()` (in `definition.py`) are internal to the module — not re-exported. `engine.py` (the `ROUTE_RUN`/`START_STAGE`/`HANDLE_AGENT_EVENT` task bodies, `attempt_promotion`, `promote_oldest_queued`, `cancel_queued`, `resume_from_pause`, `resume_with_instruction`, `resume_with_send_back`, `kill_run`, `cancel_paused`, `get_open_pause_for_run`, `resolve_send_back_target`), `boundary.py` (`evaluate_boundary`, `BoundaryDecision`), `views.py` (`list_runs_for_ticket`, `get_run_overview` — re-exported at the package level, see below), `escalation.py` (`resolve_escalation_targets`, `is_pause_responder`), `defaults.py` (the six code-shipped `PipelineDefinition` constants — `DEV`, `TROUBLESHOOT`, `IMPLEMENTATION`, `PR_REVIEW`, `INCREMENTAL_REVIEW`, `COMMENT_RESPONSE`, plus their pinned-id constants and `ALL_DEFAULTS` — see § Shipped defaults below), and `scheduler_jobs.py` (`pipeline_schedule_tick`, `resume_stalled_runs`, plus the `pipeline_schedule_tick_task`/`resume_stalled_runs_task` `TaskRef`s the `@scheduled` decorator produces) are intra-module files — `service.py`'s `start_run`/`start_rerun`/`start_rerun_from_stage`/`request_cancel`/`resolve_pause`/`instantiate_template`/`list_templates` are the sole public entry points onto the engine's write side (`defaults.py` itself is never imported outside the module); `views.py`'s two functions are the sole public entry points onto the read side. `contracts.py` (`SkillReturn`, `SkillReviewReturn`, `SkillReviewFinding`, `PriorFindingVerdict`, `bucket_confidence`, confidence-cutoff constants, `merge_prior_findings`) is intra-module — both skill contracts ride `Invocation.context["output_schema"]`, no external caller needs them. Hand-maintained JSON Schema copies of both contracts are committed at `.claude/skills/pipeline-schemas/*.schema.json` for local skill runs; `test/test_schema_files.py` fails when they drift from `model_json_schema()`.

HTTP routes: `GET/POST /api/pipelines`, `GET /api/pipelines/templates`, `POST /api/pipelines/from-template`, `GET/PUT/DELETE /api/pipelines/{id}`, `GET /api/pipelines/runs`, `GET /api/pipelines/runs/overview`, `POST /api/pipelines/runs/start`, `POST /api/pipelines/runs/{run_id}/cancel`, `POST /api/pipelines/runs/pauses/{pause_id}/respond`, `POST /api/pipelines/runs/rerun`, `POST /api/pipelines/runs/{run_id}/rerun`, `GET /api/pipelines/runs/{run_id}/stages/{stage_execution_id}/activity` — see § HTTP endpoints below.

At import time this module registers two things with lower modules it already depends on (both one-shot, at process start): the first-party `schedule` `IntakePoint` with `core/intake` (`register_intake_point`), and a pipeline-lookup callback with `domain/repos` (`register_pipeline_lookup`) — `domain/repos` can't import this module directly (it already depends on `repos` for `pipeline_referenced_by_binding`; the reverse edge would cycle), so `add_binding`'s org-ownership check and `TriggerBinding.pipeline_name` resolve through the registered callback instead. See [domain_repos.md § Cross-module pipeline lookup](domain_repos.md#cross-module-pipeline-lookup-module-boundary-note).

The reverse also happens: `domain/pr_review` (which depends on `pipelines`) registers a run-terminal hook and a comment-findings-provider callback with THIS module at ITS import time — see [domain_pr_review.md](domain_pr_review.md) and § Core user flows below.

## Module architecture

### Entities

- **Pipeline** — the stored org pipeline: a `PipelineDefinition` plus persistence metadata (`updated_at`, `updated_by_login`, `referenced`).
- **PipelineRun** — one run of a pipeline against a ticket. Pins a flattened definition snapshot at start.
- **StageExecution** — one execution attempt of one stage (or a `kind='system'` bookkeeping row — provision/cleanup/auth-refresh/push-branch).
- **RunPause** — one HITL pause; replaces `PendingHumanDecision`.

### Key value objects

- **PipelineDefinition** — the authored content (`id`, `name`, `description`, `stages`); a discriminated union on `kind` (`skill | review | action | call`). `id` (top-level and per-stage) defaults to a fresh uuid7 at parse time (`Field(default_factory=uuid7)`) — a request body that omits `id` on a new pipeline or a newly-added stage gets one server-minted for free.
- **Kickoff** — intake point + actor + input text that started a run; carries `attachment_ids: tuple[UUID, ...]` — the IDs of every `ticket_attachments` row present at `start_manual_run` time, snapshotted so later attachment changes don't affect the run. Empty for non-manual runs.
- **AttachmentRef** — frozen Pydantic VO carried in `StageInvocationContext.attachments`: `path` (`.yaaos-inputs/<filename>`), `artifact_type`, `produced_by_skill`, `role` (`"context"` | `"adopted"`), `note`. Rendered as the `## Attachments` manifest section in every skill/review invocation prompt.
- **BoundaryControl** — flat per-stage "what to do next" setting (`always_hitl | always_proceed | conditional`); `on_blocker_residuals`, `on_should_fix_residuals`, and `on_nit_residuals` gate pauses on open residuals by severity when `mode=conditional`.
- **RunOverview** — server-computed Overview-tab payload, tagged `paused | in_flight | terminal`. The `terminal` branch's `RunOutcome` carries `run_id` (the terminal run's own id) alongside `state`/`pr_url`/`failure_reason` — the SPA's rerun action on a terminal run needs the id to target `POST /runs/{run_id}/rerun`.
- **StageExecution** — the Runs-tab read-model row for one stage-execution attempt; carries `id` (the `stage_executions` row id) so the SPA can request that stage's coding-agent activity blob.

### Core user flows

- **Create/update a pipeline** — `create_pipeline`/`update_pipeline` dry-run `validate_definition()`: flattens the edited definition AND every org pipeline that transitively calls it (via `PipelineCallStage`), against a supplied org-wide id → definition map. A call cycle, an unresolvable call target, or a duplicate flattened stage name (skill/review stages only — action/call stages carry no name) raises `PipelineValidationError` before anything is written. Storage never flattens — `stages` keeps `PipelineCallStage` entries as-is, so a callee's edits reach every caller's *future* runs; the real per-run flatten is the run engine's job.
- **Delete a pipeline** — refused with `PipelineReferencedError` when `pipeline_referenced_by_call` (an app-side scan of the org's `stages` JSONB for a `call` stage targeting this pipeline) or `repos.pipeline_referenced_by_binding` (a repo trigger binding targets it) is true.
- **Read** — `get_pipeline`/`list_pipelines` resolve `updated_by_login` via `core/tenancy.get_membership_info` (the org membership handle, not a global username) and `referenced` via the same OR check delete uses.
- **Start a manual run** — `start_manual_run(ticket_id, pipeline_id, *, input_text=None, replace_in_flight=False, actor, org_id, session)` is the user-facing entry point for a manual (non-webhook) kickoff. It resolves the ticket row, calls `domain/attachments.list_attachments` to snapshot `Kickoff.attachment_ids`, builds the `Kickoff` (no PR context), and delegates to `start_run`. When `has_run_in_flight` is `True`: `replace_in_flight=False` raises `RunInFlightError` (409 at HTTP); `replace_in_flight=True` calls `request_cancel` on the in-flight run (marks it for cancellation / cancels immediately if `queued`) before starting the new one. Shape-a: caller opens the session; the function commits nothing of its own.
- **Start a run** — `start_run` accepts `triggered_by_user_id: UUID | None = None` and stamps it on `pipeline_runs.triggered_by_user_id`. Sources: the PR webhook author's linked yaaos identity (resolved via `core/identity.resolve_github_attribution` at intake time — failures write a `ticket.attribution_failed` audit row), the acting session user on rerun / re-review / batch-review endpoints, and `binding.created_by` (the user who created the schedule binding) on scheduler-fired runs; `None` for system-actor runs with no attributed user. `start_run` flattens the pipeline against *current* org definitions and pins the result as `pipeline_runs.definition_snapshot`. The row always inserts `state='queued'` first; `engine.attempt_promotion` then attempts a conditional `queued -> running` flip guarded by `ux_pipeline_runs_one_in_flight` (SAVEPOINT-wrapped — an `IntegrityError` from a concurrent promotion on the same ticket leaves the run `queued`, never raises). A successful promotion stamps `tickets.current_run_id`, flips the ticket `pending -> running` (`transition_ticket_on_run_start`), writes `run.started`, publishes `run_state_changed`, and enqueues the bootstrap `ROUTE_RUN`.
- **Run a stage** — `ROUTE_RUN` decides the next boundary action (dispatch stage 0, dispatch the next stage, or wind down to a terminal state); `START_STAGE` executes one stage and reports back to `ROUTE_RUN` (directly for `action`; via `HANDLE_AGENT_EVENT` for `skill`/`review`, once the agent's terminal event arrives). `kind='action'` stages run synchronously: `Action.execute` runs inside a SAVEPOINT (the `stage_executions` row's `status`/`action_result`/`failure_reason` land atomically with the action's own writes). `kind='skill'`/`kind='review'` stages provision a workspace first when the run doesn't have one yet (`run.workspace_id IS NULL` → a `kind='system'` `provision-workspace` stage execution dispatches and parks; on success, when `Kickoff.attachment_ids` is non-empty, a `kind='system'` `seed-inputs` stage dispatches a `WriteFilesCommand` materialising each attachment at `.yaaos-inputs/<filename>` plus a `.yaaos-inputs/.gitignore`; on its terminal success the real stage dispatch resumes at `run.current_stage_index`). An action-only pipeline never provisions, so it never runs cleanup or seed-inputs either — the permanent behavior for zero-skill pipelines, not a temporary gap.
- **Dispatch a skill stage** — `_dispatch_skill_stage` first runs `_try_adopt_for_stage`: it calls `domain/attachments.latest_matching(ticket_id, skill_name=stage.skill_name, attachment_ids=kickoff.attachment_ids)` and, when a matching attachment exists that is newer than the stage's own latest final artifact (or the stage has no prior artifact), adopts it — if `candidate.skill_version` is set and differs from the shipped skill's `version` frontmatter key (`core/coding_agent.get_shipped_skill_version`), a `stage.adoption.skill_version_mismatch` warning is logged but adoption proceeds; the body is stored via `domain/artifacts.store(adopted_from_attachment_id=attachment.id)`, marked final, a `stage.adopted` audit row written, `stage_executions.confidence` left NULL, then routing directly to the review or boundary pass (review dispatched if `stage.review` is configured; `_settle_stage_boundary` called immediately if not). When no matching attachment is found (or the existing artifact is already newer), the normal skill invocation path runs: `command_id` is minted (needed for the skill's `artifact_path = "$TMPDIR/<command_id>.md"`); `StageInvocationContext.input` is resolved (first stage: the kickoff's `input_text`; else the nearest upstream `SkillStage`'s latest final artifact body, walking backward, skipping `ActionStage`/`ReviewSkillStage` — neither produces one; falls back to the kickoff input if none found); `StageInvocationContext.attachments` is resolved (one `AttachmentRef` per `Kickoff.attachment_ids` entry — fetched via `domain/attachments.get_attachment`; already-adopted attachment ids are marked `role="adopted"` rather than `role="context"` via `adopted_attachment_ids_for_run`; the rendered prompt carries an `## Attachments` manifest listing each file's path, type, skill, role, and note); the `Invocation` is built (with `SkillReturn.model_json_schema()` injected as `context["output_schema"]`); `core/coding_agent.dispatch_invocation` is called — parking on `pending_agent_command_id`. `HANDLE_AGENT_EVENT` resumes on the terminal event: infra failures (non-`success` `outcome_label`) fail the stage with the reported reason; a `SkillReturn` schema violation or `outcome="cannot_complete"` fails the stage; `outcome="send_back"` routes through automatic send-back (§ below) instead of failing outright, unless the target is unresolvable (loud stage failure); a `completed` outcome without an artifact body fails the stage (artifact required iff `outcome == "completed"`); otherwise the artifact is stored via `domain/artifacts.store` and immediately marked final via `mark_final` (nothing reads a mid-loop artifact downstream, so marking every produced version final is harmless independent of the boundary outcome), the artifact id + raw confidence + `paths_affected` land in `loop_state`, `stage_executions.confidence` is bucketed (`bucket_confidence`) from the last **main**-skill `SkillReturn` (a review pass's confidence lands only in `loop_state`). When `stage.review` is `None` this calls `_settle_stage_boundary` (below) directly; when configured, the review-fix loop takes over first and calls it once the loop stops.
- **Assemble `PRContext`** — `engine._build_pr_context` populates `StageInvocationContext.pr` for every skill/review invocation (main, fix, review-loop pass, standalone review stage): `None` when the ticket has no bound PR or THIS run's own `Kickoff.pr_head_sha`/`pr_base_sha` weren't pinned (e.g. a non-PR-triggered run on a PR ticket); otherwise `head_sha`/`base_sha` come from the run's own kickoff pins (never the PR row's own, possibly-stale, fields) and `prev_reviewed_head_sha` is derived by `engine._prev_reviewed_head_sha` — the `pr_head_sha` of the ticket's last `completed` run that itself carried a PR kickoff (newest-`completed_at`-first, excluding the current run), `None` on the ticket's first PR review. No dedicated column — read straight off each candidate run's own `kickoff` JSONB.
- **Assemble `ActionContext`** — `engine._build_action_context` resolves `pr_external_id` from `ticket.pr_id` (`None` when unbound), and — for any action stage past the first (`stage_index > 0`) — reads the immediately preceding stage's execution row (`engine._latest_stage_execution`, latest attempt at `stage_index - 1`): `preceding_residuals` is that execution's own findings still `open` (`findings.list_for_stage_execution` filtered), `preceding_verdicts` is the last `review`-phase `loop_state` entry's recorded verdicts (`engine._last_review_verdicts` — each review pass appends its `PriorFindingVerdict`s alongside `new_finding_ids`/`residual_finding_ids`), and `preceding_artifact_id` is `engine._last_artifact_id` when the preceding stage is `kind='skill'` (`None` for a `review`-kind preceding stage, which never produces an artifact). Empty/`None` when `stage_index == 0` (an action-first pipeline) or no row exists at that index.
- **Review-fix loop** — a `SkillStage` with `review` configured runs `main → (review → fix)* → boundary` inside the *same* `stage_executions` row (`phase` cycles `main → review → fix → review → …`; `iteration` counts review passes, bumped by `_dispatch_review_invocation` before each one). The review dispatch speaks `SkillReviewReturn` over the artifact just produced, with `prior_findings` = this stage execution's own findings (any status) ∪ the ticket's other open durable findings ∪ — for a comment-response run only — findings referenced by its claimed comment batch regardless of status (`_resolve_prior_findings`, unified rule in `contracts.merge_prior_findings`; the batch's finding ids come from a registered `domain/pr_review` callback — see `register_comment_findings_provider` below). On return, `_handle_review_return` stamps `stage_executions.confidence` from `bucket_confidence(review_return.confidence)` iff `stage_executions.confidence IS NULL` at return time — allowing an adopted stage (which starts with `confidence=NULL`) to gain a confidence value from the review without overwriting a main-skill-derived confidence. Every `SkillReviewFinding` materializes via `findings.record_findings` (engine-minted uuid7 ids; the finding's skill-reported `category` — one lowercase word, pattern-constrained in the contract — becomes its `display_prefix`, so handles read `sec-001`/`arch-002` and review type is queryable per finding), then the verdict matrix applies mechanically for each `PriorFindingVerdict` (`_apply_verdicts`): `fixed → findings.resolve`; `still_present → findings.reflag` when the referenced finding is open, `findings.reopen` when resolved; `user_overrode → findings.dismiss`. Residuals = this stage execution's still-`open` findings, recorded on the `review`-phase `loop_state` entry every pass. While residuals remain and `iteration < review.max_iterations`, `_dispatch_fix_invocation` reruns the main skill with `revision=RevisionContext(source="fix", text=<rendered residuals>, prior_artifact=<last artifact body>)` — the rendered text + `source` are also stamped onto `stage_executions.revision` (independent of the wire payload, so the fix's input is durably observable) — then re-reviews the new artifact. The loop stops once residuals are empty or the iteration cap is hit, then `_settle_stage_boundary` runs. A standalone `ReviewSkillStage` (`kind='review'`) dispatches exactly one `SkillReviewReturn` invocation the same way (findings, verdicts, residuals) but stores no artifact and never loops, then calls `_settle_stage_boundary` with `paths_affected=()` (`SkillReviewReturn` carries no such field, so `on_protected_code` can never trip for it) — `_dispatch_review_only_stage` / `_handle_review_stage_event`.
- **Evaluate a stage boundary** — `_settle_stage_boundary` computes residuals (this stage execution's still-`open` findings — empty for a review-off stage), checks them for a resolvable `defect_in_artifact` send-back FIRST (see below — an unresolvable name degrades to a plain residual and falls through), then fetches the ticket's `repo_external_id` and calls `boundary.evaluate_boundary(stage.boundary, residuals=…, paths_affected=…, confidence=stage_executions.confidence, …)`. `confidence` is `Confidence | None` — NULL for adopted stages that either had no review or whose review returned no confidence. `evaluate_boundary` skips the `on_confidence_below` check entirely when `confidence is None` (no main-skill ran, so no bucket to compare against). A main-skill `cannot_complete` outcome never reaches it (fails the stage earlier); a resolvable main-skill `send_back` outcome never reaches it either (routes straight to send-back, before any artifact or residual exists) — so by the time `evaluate_boundary` runs, the stage's own work has always "completed". `always_proceed`/`always_hitl` short-circuit; `conditional` accumulates every tripped condition (`on_blocker_residuals`/`on_should_fix_residuals`/`on_nit_residuals` against residual severities, `on_protected_code` against `repos.evaluate_protected(paths_affected)`, `on_confidence_below` against the bucketed confidence — skipped when `confidence is None`) rather than stopping at the first hit. `proceed` marks the stage `completed`/`boundary_outcome="proceeded"` and enqueues `ROUTE_RUN`; `pause` calls `_enter_pause`.
- **Enter a pause** — `_enter_pause` marks the stage `boundary_outcome="paused"` (`boundary_detail` = the tripped conditions), resolves the escalation set (`escalation.resolve_escalation_targets` unioned with `BoundaryDecision.protected_owner_user_ids` when `on_protected_code` fired), inserts an open `run_pauses` row, flips `pipeline_runs.state='paused'`, extends the run's workspace expiry by `PAUSED_RUN_WORKSPACE_GRACE_SECONDS` (`core/workspace.extend_expiry`) when one exists, flips the ticket to `hitl` (`transition_ticket_on_run_paused`), publishes `run_state_changed`, writes `run.paused`, and fans out a `pipeline_run_paused` notification (`core/notifications.create`) to every resolved escalation target.
- **Resolve a pause** — `resolve_pause` (in `service.py`) loads the `run_pauses` row (org-scoped), 403s via `NotEscalationTargetError` unless the actor is in `escalation_user_ids` or an org admin/owner (`escalation.is_pause_responder`), 409s via `PauseAlreadyResolvedError` if already resolved, and raises `InvalidPauseResolutionError` (400 at HTTP) for a resolution missing its required field (`instruct` without `instruction`, `send_back` without `send_back_to_stage`) or a `send_back_to_stage` that isn't an upstream `SkillStage` in the paused stage's own flattened definition (validated via `engine.resolve_send_back_target` before anything is written). `approve` calls `engine.resume_from_pause`: flips the run back to `running`, the ticket back to `running` (`transition_ticket_on_run_resumed`), and enqueues `ROUTE_RUN(completed_stage_index=<the paused stage>, outcome_label="success")` — the exact call the stage would have made itself had the boundary not tripped. `instruct` calls `engine.resume_with_instruction`: a NEW `stage_executions` row at the SAME `stage_index`, dispatched via `_start_stage_impl` (workspace-liveness-checked, re-provisioning if needed) with `RevisionContext(source="instruction", text=<the human's text>, prior_artifact=<the stage's own latest final artifact, or "">)`. `send_back` calls `engine.resume_with_send_back` — the same loop-protected rewind machinery an automatic send-back uses (see below), human-sourced. `kill` calls `engine.kill_run`, which routes through the normal terminal path (`_finish_or_cleanup(… "killed" …)`) — same cleanup-first routing as any other decided outcome. Every resolution writes `run.pause_resolved` before applying its effect.
- **Automatic send-back** — the boundary order (per architecture) is: send-back first, then `cannot_complete`, then mode/conditions. A main/fix `SkillReturn.outcome == "send_back"` is validated at return time (`_validate_skill_return_and_artifact`) against `engine.resolve_send_back_target` — an earlier `SkillStage` in the flattened definition (skill stages are the only artifact-producing kind a send-back can target), respecting the stage's own `context_stages` restriction when set. An unresolvable target is a loud stage (and run) failure — a send-back that can't resolve must not silently proceed. A residual finding's `defect_in_artifact` is checked at boundary-settlement time (`_settle_stage_boundary`, before `evaluate_boundary` ever runs) via the same resolution helper; an unresolvable name degrades to a plain residual (logged, not a contract violation — the attribution is best-effort). Either path calls `_send_back_to_stage`: `run.sendback_counts[target_stage_name]` (bare key — distinct from the `auth_retry:{stage}` keys the auth-expired retry uses) gates the rewind — a target already sent back to once THIS run pauses instead (`tripped={"sendback_loop": target_stage_name}`) rather than rewinding a second time. A successful rewind marks the sending stage `boundary_outcome="sent_back"`, builds `RevisionContext(source="send_back", text=<the gap — the skill's own `outcome_reason`, or the finding's `body`>, prior_artifact=<the TARGET stage's own latest final artifact>)`, and dispatches the target fresh via `_start_stage_impl` (same workspace-liveness/re-provision safety net as any forward dispatch) — a NEW `stage_executions` row at the target's index. The run then re-runs FORWARD through the intervening stages once the target re-completes (normal `ROUTE_RUN` routing).
- **Re-entry revision threading across an async provision hop** — `instruct`/`send_back`/rerun all need to hand a `RevisionContext` to a dispatch that may first require re-provisioning the workspace (async — parks awaiting the agent). `Kickoff.revision` (a single-use field, `None` on every ordinary run) carries it across that hop: the re-entry path stashes the revision on `run.kickoff` before calling `_start_stage_impl`; `_start_stage_impl` reads `kickoff.revision`, clears it (rewrites `run.kickoff`) so it's consumed exactly once, and threads it into `_dispatch_skill_stage`/`_dispatch_review_only_stage`'s `revision` parameter — durably stamped onto the new `stage_executions.revision` column (same idiom the fix-loop already uses) independent of the wire payload.
- **Reach a terminal state** — once `ROUTE_RUN` decides a target state (`completed`/`failed`/`cancelled`/`killed`), `_finish_or_cleanup` dispatches the `cleanup-workspace` system stage first when the run ever provisioned a workspace (`run.workspace_id IS NOT NULL`); `_finalize_after_cleanup` re-derives the same target state once cleanup's terminal event arrives (cleanup's own failure is logged, not re-surfaced — the run's outcome was already decided) and calls `_enter_terminal`. A run that never provisioned (action-only) enters the terminal state directly. `_enter_terminal` writes the terminal `pipeline_runs.state`, flips the ticket to the matching status (`transition_ticket_on_run_terminal`), writes `run.{state}`, publishes `run_state_changed`, notifies the resolved escalation targets on `failed` (`escalation.resolve_escalation_targets` — kickoff actor, else schedule notify list, else the PR author's linked identity, else the org's admins), enqueues every hook in the run-terminal hook registry (below), and promotes the oldest `queued` run on the ticket (`promote_oldest_queued`).
- **Run-terminal hooks** — `register_run_terminal_hook(task_ref)` (list-shaped, mirrors `core/agent_gateway`'s consumer registry) lets another module react to every pipeline run reaching a terminal state, without `domain/pipelines` importing it. `_enter_terminal` enqueues each registered hook (`{org_id, ticket_id}` args) inside the SAME transaction that flips the run terminal. `domain/pr_review` registers its `AFTER_RUN_TERMINAL` task at import time (comment-batch re-evaluation once the ticket frees up).
- **Comment-findings provider** — `register_comment_findings_provider(fn)` is the cycle-avoidance callback `domain/pr_review` registers at import time (it already depends on `pipelines`; the reverse edge would cycle) so `_resolve_prior_findings` can pull a comment-response run's claimed-batch finding ids without `pipelines` importing `pr_review`. Mirrors `domain/repos.register_pipeline_lookup`'s shape.
- **Cancel a run** — `request_cancel`: `queued` cancels immediately (`engine.cancel_queued`); `running` sets `cancel_requested` and the next `ROUTE_RUN` boundary check routes to `cancelled` instead of the next stage — checked on every boundary including the last one, so a run cannot slip into `completed` after cancellation was requested; `paused` cancels immediately too (`engine.cancel_paused` — closes the open pause row with `resolved_at` stamped and no `resolution` recorded, then routes through the same cleanup-first terminal path). Raises `RunAlreadyTerminalError` on an already-terminal run (409 at HTTP).
- **Fire a schedule-kind trigger binding** — `pipeline_schedule_tick` (`* * * * *`, `queue="default"`) reads `repos.list_due_schedule_bindings(now)` once, then per due binding opens `org_context` and creates the ticket + starts the run in ONE transaction (`_fire_one`): `tickets.create_from_schedule` (`source="schedule"`, `source_external_id="{binding_id}:{fire_time}"` — the tickets `(org_id, source, source_external_id)` unique constraint makes a redelivered slot a no-op, title = `Schedule.name`, `branch_name` freshly minted) then, iff the ticket was actually created, `service.start_run(Kickoff(...), triggered_by_user_id=binding.created_by)` — the binding's creator (nullable) becomes the run attribution; `Actor.system()` is still the kickoff actor so escalation routes to `notify_user_ids` rather than a single user (`escalation.resolve_escalation_targets`'s resolution order). The VCS plugin for the binding's `repo_external_id` is resolved via the module-private `_resolve_plugin_id`, which delegates to `core/vcs.resolve_plugin_id_for_repo` (single-plugin fast path; `is_repo_accessible` disambiguation for multiple plugins) — a schedule binding carries no plugin id of its own (unlike GitHub's namespaced intake points). A single binding's failure is logged and skipped so it never blocks the other due bindings in the same slot; cluster safety for the tick itself is `core/tasks`' own per-slot claim, not this module's.
- **Stall sweep — three idempotent reconciliations** — `resume_stalled_runs` (`* * * * *`) makes Redis fully disposable: **(a)** a `running` run with `pending_agent_command_id IS NULL` and `updated_at` older than `Settings.yaaos_run_stall_threshold_seconds` (default 300s) has its next-dispatch message re-derived from the run's own latest non-`system` `stage_executions` row (none yet → bootstrap `ROUTE_RUN(completed_stage_index=None)`; else replay `ROUTE_RUN(completed_stage_index=<that row's stage_index>, outcome_label="success"|"failure")`) — this reproduces exactly what the lost message would have done, because `_dispatch_stage`'s own `current_stage_index` update is idempotent and the recomputed `next_index` matches regardless of which side of the lost message the sweep catches it on. **(b)** a `running` run whose `pending_agent_command_id` points at an `agent_commands` row already `status="done"` (`core/agent_gateway.get_command_status`) — the terminal event WAS recorded but the resume never processed (outbox message dispatched then lost) — is re-routed via a synthetic `HANDLE_AGENT_EVENT(outcome_label="failure")`; the original outputs aren't recoverable (`agent_commands` carries no output payload), so the replay is a loud, conservative failure rather than a fabricated success. Gated on the same staleness threshold as (a) so a synthetic resume never races a real one still in flight — `engine._handle_agent_event_impl`'s stale-command-id guard makes whichever resume processes first authoritative regardless of ordering. **(c)** a ticket with a `queued` run and no `running`/`paused` sibling (`promote_oldest_queued`) — backstop for the promotion race `attempt_promotion` already guards at the transaction level.
- **Shipped defaults** — `defaults.ALL_DEFAULTS` is six frozen, module-level `PipelineDefinition` constants with pinned uuid7 ids (never persisted; a new org starts empty). Composition: `dev` = `requirements` (review = `pipeline-requirements-review`, `max_iterations=2`) → `architecture` (review = `pipeline-architecture-review`, `max_iterations=2`) → `plan` → call `implementation`; `troubleshoot` = `diagnose` → `fix-plan` (stage name; runs the `plan` skill — stage names and skill names are independent) → call `implementation`; `implementation` = `implement` (review = `pipeline-code-review` — code, security, and architecture categories — `max_iterations=3`) → `github:create_pr`; `PR review` = `code-review` (standalone `ReviewSkillStage` running `pipeline-code-review`) → `github:update_pr`; `incremental review` = the same shape → `github:update_pr` (the diff-since-last-review behavior is `engine._build_pr_context`'s `prev_reviewed_head_sha` derivation, not a per-stage config difference — the two review pipelines are otherwise identical); `comment response` = `comment-response` (standalone `ReviewSkillStage`) → `github:reply_to_comment`. Skill names are pinned repo files under `.claude/skills/<name>/SKILL.md`, all `pipeline-`-prefixed (run-engine consumption, not human invocation): `pipeline-requirements`, `pipeline-requirements-review`, `pipeline-architecture`, `pipeline-architecture-review`, `pipeline-plan`, `pipeline-implement`, `pipeline-code-review`, `pipeline-diagnose`, `pipeline-comment-response`. The `plan` stage skill (`pipeline-plan`) produces a PhaseBlock sequence — an ordered set of self-contained per-phase briefs that `pipeline-implement` executes one at a time in a fresh-context subagent; each block is the complete contract for that phase (verified cites, Changes per file, Acceptance, Verification, Tests added). `list_templates()` returns `defaults.ALL_DEFAULTS` verbatim. `instantiate_template(org_id, template_id, actor, session)` deep-copies the requested template into a fresh org `pipelines` row (fresh pipeline + stage ids via `service._materialize_template`) and rewires every `PipelineCallStage` it contains, depth-first: a callee already present in the org (matched by name) is reused; otherwise it's copied too, recursively (memoized within one call so a callee reached twice is only materialized once). Only the top-level requested template is unconditionally fresh-copied — the by-name reuse rule applies to callees only, so instantiating `dev` then `troubleshoot` into the same org shares one `implementation` copy between them. Raises `TemplateNotFoundError` for an unknown id. `dev` and `troubleshoot` carry no shipped intake binding (no per-run input a webhook/schedule can supply) — they're exercised by driving `instantiate_template` + `start_run` directly, not through a Repos-page trigger.
- **Read a stage's coding-agent activity** — the Runs tab's per-stage activity endpoint reuses `core/coding_agent.get_stage_activity`, keyed on `(coding_agent_runs.run_id, stage_execution_id)` (both UUID columns); the pipelines engine stamps `run_id=<pipeline_run_id>` and `stage_execution_id=<stage_execution_id>` at every `coding_agent.dispatch_invocation` call. 404s when the run doesn't belong to the caller's org.
- **Re-run a failed/cancelled/killed run from the top** — `start_rerun` (`org_id`, `run_id`, `actor`) is a NEW run on the SAME ticket — approximate-resume semantics: nothing is inherited from the old run (no artifact read-through, no pinned snapshot reuse); continuity comes entirely from durable state the run's failure left behind (branch commits, ticket findings, a bound PR). Raises `RunNotRerunnableError` unless `run_id` is the ticket's current run (`tickets.current_run_id` — set at every promotion, never cleared at terminal, so a superseded run's own id fails this check even though its own state is still `failed`) AND its state is one of `failed`/`cancelled`/`killed`. Clones the prior run's `Kickoff` (`model_copy(update={"actor": actor, "intake_point_id": "rerun", "revision": None})` — preserves `input_text`/`pr_base_sha`/`pr_head_sha`/`notify_user_ids`, replaces the actor with the caller, clears any stale single-use revision), then delegates entirely to `start_run` against the run's own `pipeline_id` — re-flattened against CURRENT org definitions, at stage index 0. Raises `PipelineNotFoundError` if the pipeline was deleted since. Distinct from `start_rerun_from_stage` (continues from an arbitrary stage, reads earlier artifacts through) — the two coexist for different intents.
- **Instruct & re-run from a completed/failed run** — `start_rerun_from_stage` (`org_id`, `ticket_id`, `from_stage`, `instruction`, `actor`) starts a NEW run rather than resuming the old one. It resolves the CURRENT definition via the ticket's `current_run_id`'s `pipeline_id` (re-flattened fresh — a since-edited pipeline is what actually runs, not the old run's pinned snapshot), raising `RunNotFoundError` if the ticket has no prior run. `from_stage` must resolve to a `SkillStage`/`ReviewSkillStage` in the fresh flatten, else `StageNotInDefinitionError` (409). Before starting, `_nearest_upstream_skill_stage_resolvable` mirrors `_resolve_stage_input`'s own backward walk: if the nearest upstream `SkillStage` (or the next one back, etc.) doesn't resolve via `artifacts.latest_final`, raise `MissingInheritedArtifactError` (409) — refuse loudly rather than start a run that silently starves on a renamed/removed stage. Earlier stages' artifacts are read through by name (no copy) — the new run's `definition_snapshot` starts fresh at `from_stage`'s index; `_resolve_stage_input`'s existing backward walk naturally finds the prior run's artifacts under the same stage names. The new row is created `state="queued"`, `current_stage_index=from_index`, with `Kickoff.revision=RevisionContext(source="instruction", text=instruction, prior_artifact=<from_stage's own prior final artifact, if any>)` — `ROUTE_RUN`'s bootstrap dispatch (`_route_run_impl`, `completed_stage_index is None`) reads `run.current_stage_index` instead of hardcoding `0`, so it jumps straight to `from_stage`; the revision then rides across the provision hop per the mechanism above. Queues (via the normal `attempt_promotion` path) if another run is already in flight on the ticket.

### State machines

- **Run state**: `queued → running ⇄ paused → completed | failed | killed | cancelled`. `paused` is entered by `_enter_pause` when a stage's `evaluate_boundary` call decides `pause`; `resolve_pause(approve)` flips back to `running`, `resolve_pause(kill)` or `request_cancel` on a `paused` run flips to a terminal state.
- **Run phase**: `provision → stages → cleanup`. `'provision'` while a workspace provision (initial or re-provision) is in flight; `'stages'` for normal stage dispatch/execution (this includes the `seed-inputs` system stage, which runs synchronously between provision success and the first skill dispatch — the phase stays `'stages'` throughout); `'cleanup'` while the terminal-bound cleanup system stage is in flight. An action-only run never leaves `'stages'` — see § Core user flows.
- **Workspace liveness check.** Before every skill/review-stage dispatch, `_workspace_is_live` calls `core/workspace.get_workspace_info` and checks `status == ACTIVE`. A dead or absent workspace (any reason: reaper, idle timeout, agent restart, pause-grace elapsed) triggers `_dispatch_provision_stage`, which overwrites `run.workspace_id` with a new id and runs a fresh `provision-workspace` system row. Mid-run re-provision is structurally identical to the initial provision from the engine's perspective.
- **Auth-expired retry recovery.** An `auth_expired` failure on a skill stage's `main` phase inserts a `refresh-auth` system row, dispatches `core/workspace.dispatch_auth_refresh`, and parks. On refresh-auth success the skill invocation is retried once (new `stage_executions` row). The one-retry cap is tracked in `run.sendback_counts` under `auth_retry:{stage_name}` (same map as sendback tracking, distinct key namespace). A second `auth_expired` on the retry exhausts the cap and fails the run normally. An `auth_expired` failure during the `review`/`fix` phases (or on a standalone `review`-kind stage) is treated as a plain infra failure instead — the resume path only knows how to restart a stage fresh at `main`, and doing that mid-loop would silently discard `loop_state` and the iteration count.
- **Send-back loop protection.** `run.sendback_counts[target_stage_name]` (bare stage name, no prefix) tracks how many times THIS run has rewound to a given target. The first send-back to a target rewinds; a second send-back to the SAME target this run pauses instead (`tripped={"sendback_loop": target_stage_name}`) rather than rewinding forever.
- **`PAUSED_RUN_WORKSPACE_GRACE_SECONDS = 1800`** — module constant for the workspace-expiry extension `_enter_pause` grants via `core/workspace.extend_expiry`; the normal reaper collects the workspace once the deadline passes with no resolution.

### Deferred (not yet implemented)

- **`dispatch_push`** exists as a plain function (push-failure recovery after auth refresh) but has no caller yet — no stage currently pushes a branch from this engine.

## HTTP endpoints

`domain/pipelines/web.py`, prefix `/api/pipelines`, all `ORG_SCOPED`. Definition CRUD is gated by `Action.PIPELINES_MANAGE` (admin minimum); the run-read, run-cancel, pause-respond, and rerun endpoints are gated by `Action.REVIEWER_WRITE`/`Action.REVIEWER_READ` (builder minimum) — a run being watched on (or rewound from) a ticket is a builder-facing concern, not a pipeline-authoring one. `respond`'s actual authorization is dynamic on top of that floor: the resolved escalation set union org admins/owners (`NotEscalationTargetError` → 403 otherwise). `GET /runs` and `GET /templates` are both registered BEFORE `GET /{pipeline_id}` — the latter is a bare single-segment match (route matching precedes FastAPI's own UUID parsing) and would otherwise swallow the literal `/runs` or `/templates` path.

## Data owned

- `pipelines` — org-scoped pipeline definitions. `UNIQUE(org_id, name)`. `id` has no DB-level default (unlike its sibling tables below) — the pipeline's own id participates in `PipelineCallStage.pipeline_id` and templates ship with pinned ids, so `create_pipeline` mints it app-side via the definition model's `id` field (itself defaulting to a fresh uuid7 when the request omits it). See `apps/backend/docs/patterns.md` § UUID primary keys.
- `pipeline_runs` — one row per run. `triggered_by_user_id UUID NULL FK users ON DELETE SET NULL` — the yaaos user attributed to the run (PR author, rerun actor, schedule-binding creator); `NULL` for system-actor runs with no attributed user. `UNIQUE INDEX ux_pipeline_runs_one_in_flight ON (ticket_id) WHERE state IN ('running','paused')` enforces one in-flight run per ticket at the DB layer; the run engine's promotion mechanics are built directly on this index.
- `stage_executions` — one row per stage-execution attempt. `CHECK` constraints on `kind`, `status`, `phase`, `confidence`, `boundary_outcome`. `action_result` is the `Action.Result` dump for `kind='action'` rows. `kind='system'` rows (`provision-workspace`, `seed-inputs`, `cleanup-workspace`, `refresh-auth`) carry `stage_index=NULL`. `kind='skill'` rows carry `loop_state` entries per pass (`main`/`review`/`fix`, each with the produced artifact's id + raw confidence, or the review pass's new/residual finding ids) and `confidence` bucketed from the last main-skill pass onto the row; `revision` carries the last fix dispatch's `RevisionContext` dump (`source`, rendered `text`, `prior_artifact`). `kind='review'` rows (standalone `ReviewSkillStage`) carry one `review`-phase `loop_state` entry and no artifact.
- `run_pauses` — one row per HITL pause. `escalation_user_ids` is a `UUID[]`; `tripped` records which conditions fired (`{"always_hitl": true}`, a `conditional` mode's fired keys, or `{"sendback_loop": <target_stage_name>}` when a second send-back to the same target pauses instead of rewinding); `resolution`/`instruction`/`send_back_to_stage`/`resolved_by`/`resolved_at` are null until `resolve_pause` applies. A cancelled-out-from-under pause has `resolved_at` set but `resolution` null.
