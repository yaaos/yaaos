# core/workspace

> Provisioned environments for code work — DB-backed lifecycle, plugin actuators, and a reaper.

## Scope

- **Owns:** `Workspace` + `WorkspaceProvider` Protocols, `WorkspaceRegistry` (ContextVar-bound provider map), `workspaces` table lifecycle, per-minute reaper (`@scheduled` task in [`core/tasks`](core_tasks.md), cron `* * * * *`), single-flight claim registry, the plain dispatch functions covering workspace lifecycle (`dispatch_provision`, `dispatch_cleanup`, `dispatch_auth_refresh`, `dispatch_push`). Implements and registers `WorkspaceAgentReportSink` (the IoC seam to [`core/agent_gateway`](core_agent_gateway.md)).
- **Does not own:** lifecycle *policy* (that's callers); workspace filesystem internals (plugin-private); `domain/tickets` data.
- **Receives:** `WorkspaceSpec` from callers; AgentEvent ingestion goes through the registered sink. **Emits:** `workspace.transitioned` audit rows via [`core/audit_log`](core_audit_log.md); AgentCommand dispatches to [`core/agent_gateway`](core_agent_gateway.md).

## Why / invariants

- **`close_workspace` does not call `provider.destroy()` synchronously** — keeps close fast; all destroy retries flow through the reaper. Only the reaper destroys. `close_workspace` is atomically idempotent under concurrent admin requests and reaper transitions — a single conditional `UPDATE … WHERE status='active' RETURNING org_id` ensures at most one audit row per real ACTIVE → EXPIRED transition.
- **`_attempt_destroy` is single-flight per cycle.** The EXPIRED→DESTROYING UPDATE narrows by `WHERE status='expired'` so exactly one concurrent reaper cycle can win the transition. A caller that sees `rowcount==0` exits immediately — no audit row, no `provider.destroy()` call. `destroy_attempts` is incremented at the SQL layer (`destroy_attempts = destroy_attempts + 1`); the new value is captured via `RETURNING` so the catch-block failure-cap check never reads a stale Python snapshot.
- **`on_stream_line` callback** — when provided, the provider reads stdout line-by-line (live JSON parsing); when absent, buffers to completion. Timeout/cancel paths are unchanged either way.
- **Each new workspace capability is a deliberate named method** (`run_coding_agent_cli`, `read_text`). A generic `exec(argv)` would silently broaden.
- **Run correlation is via `agent_commands.run_id`**, not the workspace row. `try_claim` writes only `current_command_id` and (if not yet set) `owning_agent_id`; `release_claim` clears `current_command_id`. No per-workspace `current_holder_run_id` column exists.
- **Workspaces are hard-tied to their owning agent.** The agent that claims and executes `ProvisionWorkspace` owns the workspace for its whole life. Post-provision commands (cleanup, etc.) are enqueued with `agent_id` pre-stamped via `pin_command_to_agent` so `claim_next`'s `workspace_ids` sweep routes them to the same agent. In-memory/legacy rows keep `agent_id` NULL. The owning agent's id is what `core/agent_gateway` reads to authorize that agent's workspace/command-event posts (see [`core/agent_gateway`](core_agent_gateway.md)).
- **`dispatch_provision_workspace` no longer pre-assigns an agent.** The command is enqueued as `pending` with no `agent_id`; whichever reachable agent has capacity claims it via `claim_next`'s `new_workspaces` pull. The agent that executes the command becomes the workspace owner via the `WorkspaceRow.agent_id` stamp at `try_claim`.
- **`dispatch_via_workspace` is Layer 2.** Loads the workspace row (for `org_id` + `owning_agent_id`), calls `enqueue_command` (Layer 1), pins the command to the owning agent when one is set, and — when `claim_workspace=True` — atomically claims via `try_claim`. Raises `WorkspaceNotFoundError` when no row exists; raises `WorkspaceClaimFailed` when `claim_workspace=True` but the row is already claimed or inactive. All workspace dispatch helpers except `dispatch_provision` (which has no row yet) route through this function. `core/coding_agent.dispatch_invocation` (Layer 3) calls it with `claim_workspace=True`; `dispatch_cleanup` and `dispatch_auth_refresh` call it with `claim_workspace=False`.
- **Plain dispatch functions in `dispatch.py`** are the sole workspace-lifecycle dispatch surface; `domain/pipelines`'s run engine is the shipped caller:
  - `dispatch_provision(spec: ProvisionWorkspaceSpec, ctx, *, session) -> UUID` — `ProvisionWorkspaceSpec` carries `workspace_id` (caller-minted; no row exists yet to derive it from), `org_id`, `plugin_id`, `repo_external_id`, and the checkout instruction (`head_sha` / `branch_name` / `base_sha`). Fetches install credentials via `core/vcs.get_install_credentials` and enqueues via `dispatch_provision_workspace` (Layer 1).
  - `dispatch_cleanup(workspace_id, ctx, *, session) -> UUID` — enqueues `CleanupWorkspaceCommand`; never claims.
  - `dispatch_auth_refresh(workspace_id, ctx, *, org_id, plugin_id, session) -> UUID` — mints a fresh installation token via `core/vcs.get_installation_token(plugin_id, org_id)` and enqueues a `RefreshWorkspaceAuthCommand` carrying it; never claims. `domain/pipelines`' run engine calls this directly on an `auth_expired` skill-stage failure (passing `run.org_id` + the ticket's `plugin_id`), then retries the stage once.
  - `dispatch_push(workspace_id, ctx, *, session) -> UUID` — enqueues `PushBranchCommand` (push-failure recovery only); never claims.
  - All four take `ctx: agent_gateway.DispatchContext` (`run_id`, `ticket_id`, `stage_execution_id`, `attempt`, `traceparent`); `dispatch_auth_refresh` additionally takes `org_id`/`plugin_id` to source a fresh token.
- **Failsafe-6 is per-agent.** Workspaces whose owning agent newly transitioned to offline (from `compute_agent_liveness_transitions`) are expired with reason `agent_loss` on the same reaper tick; only that agent instance's bearers are revoked. The reaper passes the sweeper's newly-offline set directly to `failsafe_agent_loss` — no second `stale_agent_ids` probe. The DELETE handler also calls `failsafe_agent_loss` eagerly via `WorkspaceAgentReportSink.handle_agent_loss` for immediate cleanup. Workspaces with NULL `owning_agent_id` are skipped.
- **Terminal-failure synthesis on agent loss.** For each expired workspace with a non-null `current_command_id`, `failsafe_agent_loss` resolves the run_id from `agent_commands.run_id`, then calls `agent_gateway.enqueue_agent_event(outcome_label="failure", ...)` — the same agent-event consumer registry `record_agent_event` uses (see [core_agent_gateway.md § Agent-event consumer registry](core_agent_gateway.md#agent-event-consumer-registry)) — so the run engine resumes rather than waiting forever. If `get_command_run_id` returns None (no command row or no correlation), the stage hangs until the run's stall sweeper fires — the same pre-existing behaviour.
- **`WorkspaceAgentReportSink` is registered at import time** (`workspace/__init__.py`). agent_gateway's service functions call the sink for workspace-state reads/writes; the sink implementation lives in `app/core/workspace/agent_report.py`. The stale-claim guard (kind mismatch or unknown workspace) returns `accepted=False`; agent_gateway maps that to `StaleClaimError`, which the endpoint returns as `410 Gone`.
- **Lean workspace row creation.** The sink inserts the `workspaces` row lazily with `status='active'` (never through `creating`), `owning_agent_id` from the reporting bearer, and `org_id` resolved from the originating `agent_commands` row (via `get_command_org_and_payload`). Two entry points share the insert: the legacy `apply_workspace_event` path (first `created`/`ready` workspace event) and `materialise_provision_success`, which `core/agent_gateway` calls on a `ProvisionWorkspace` terminal `completed_success` (the Go agent never sends workspace events). `materialise_provision_success` is idempotent — an existing row for the workspace is left untouched. The `workspace_id` is minted up front by the caller of `dispatch_provision`; the row materialises only once an agent owns it.
- **Lean row shaping is payload-derived and vendor-neutral.** `expires_at`/`max_idle_seconds` come from the command payload's `ttl_seconds`/`max_idle_seconds` (falling back to a module default only when absent); `provider_id` is resolved from the registered `WorkspaceProvider`'s `plugin_id`, not a hardcoded constant, so a second provider entering this path is labelled correctly. `spec` carries only `{"sha": <head_sha>}` — the cleartext GitHub installation token in the command payload is deliberately excluded: it is a short-lived wire credential that lives on the retired command row and must never persist in long-lived workspace state.
- **`release_command_claim` precedes routing on every terminal event.** Before the run engine is resumed (via `handle_agent_event`), the sink clears `current_command_id` so the next `try_claim` sees `NULL` — failure-report-precedes-disposal ordering. No-op when no workspace row holds the command (e.g. before the lean row exists).
- **`dispatch_provision` calls `core/vcs.get_install_credentials`** at dispatch time to build `RepoRef` + `AuthBlock`. `get_install_credentials(plugin_id, org_id, repo_external_id)` returns an `InstallCredentials` model (`clone_url`, `installation_token`). The `ProvisionWorkspaceSpec` typed model carries only `workspace_id`, `org_id`, `plugin_id`, `repo_external_id`, `head_sha`, `branch_name`, `base_sha`; no credentials touch the spec.
- **`ProvisionWorkspaceSpec` carries a checkout instruction: exactly one of `head_sha` (detached pin — PR tickets) or `branch_name` (named work branch — yaaos-authored tickets) must be set**, mirroring the same exactly-one-of contract on `agent_gateway.RepoRef`. Both flow straight through to the wire `RepoRef`. Commit identity (`git_user_name`/`git_user_email`) is not threaded through `ProvisionWorkspaceSpec` at all — it defaults to the backend-supplied constants (`DEFAULT_GIT_USER_NAME`/`DEFAULT_GIT_USER_EMAIL` in `agent_gateway.types`) on `ProvisionWorkspaceCommand` itself, so every dispatch carries it with zero caller involvement.

## Gotchas

- **Admin HTTP endpoints are unauthenticated.** Gate behind an authenticated proxy before exposing to untrusted networks.
- **`ResourceCaps` / `NetworkPolicy` are advisory** — values exist to stabilise the interface across provider implementations; the remote agent forwards them to the Go process.
- **Failsafe-5 (disk sweep) is in the Go agent** (`apps/agent/internal/supervisor`), not here.
- **Never hand-edit `tach.toml`** — run `apps/backend/bin/sync_modules` after interface changes.
- **No boot-time bulk recovery.** The reaper's heartbeat-staleness path (failsafe-6) is the recovery surface for workspaces left in non-terminal states by a crashed process — there is no lifespan hook that bulk-flips them at startup.

## Vocabulary

- **WorkspaceProvider** — dumb actuator (provision + run + destroy + health-check); no policy. Methods carry no `plugin_state` — the workspace row is the durable record. Only registered implementation is `RemoteAgentWorkspaceProvider` (`remote_agent`).
- **Reaper** — per-minute `@scheduled` task `workspace_reaper` enforcing TTL expiry, idle-timeout, agent-loss detection, and destroy retries. Cluster-safe via `core/tasks`'s per-tick atomic claim; body is idempotent (reads fresh state from the DB each sweep).
- **`dispatch_auth_refresh`** — recovery dispatch called directly by `domain/pipelines`' run engine on an `auth_expired` skill-stage failure; mints a fresh installation token and sends a `RefreshWorkspaceAuth` AgentCommand so the agent rotates credentials without tearing down the workspace, then the engine retries the failing skill stage once.

## Data owned

- `workspaces` — `(id, org_id, owning_agent_id, provider_id, spec jsonb, status, current_command_id, max_idle_seconds, created_at, activated_at, expires_at, destroyed_at, destroy_attempts, last_destroy_attempt_at, last_destroy_error)`. `owning_agent_id` is `workspace_agents.id` (NOT NULL FK, `ON DELETE RESTRICT`; set at lean row creation). RESTRICT enforces the invariant that a `workspace_agents` row cannot be deleted while any workspace still references it — a workspace never outlives its owning agent. Indexes: `(status, expires_at)`, `(org_id, created_at)`, `owning_agent_id`. CHECK `ck_workspaces_id_uuidv7` (`uuid_extract_version(id)=7`): `id` is minted app-side in `ProvisionWorkspace.dispatch` (the agent's lifecycle handle), so it must be a `uuid7()` — the constraint enforces it at the row boundary (added `NOT VALID`, grandfathering pre-guard rows). See [patterns.md § UUID primary keys](patterns.md).

## Routes

- `GET /api/workspaces/connection_status` — `ORG_SETTINGS_READ` (Admin+). Aggregated heartbeat state for the current org. Returns `{state, pod_count, latest_heartbeat_at}` where `pod_count` is the number of agent instances. State values: `connected`, `lost`, `not_configured`. Implemented in `app/core/workspace/web.py`; delegates to [`core/agent_gateway`](core_agent_gateway.md) `connection_status_for_org`.

## How it's tested

`app/core/workspace/test/test_dispatch.py` covers `try_claim` / `release_claim` contention and `owning_agent_id` stamping. `app/core/workspace/test/test_connection_status_endpoint.py` covers the HTTP route: auth enforcement (401, 403) and the `not_configured` happy path. `app/core/workspace/test/test_agent_report.py` covers the sink: kind→status map, stale-claim guard, heartbeat reconciliation, `resolve_claim` (reads `agent_commands.run_id`), owning-agent resolution. `test_reaper_failsafes.py` covers per-agent failsafe-6 and `failsafe_agent_loss` correlation via `agent_commands`. `test_lean_lifecycle_service.py` (service tier) covers lean row creation on the agent's first event, `release_claim` timing, and the `ProvisionWorkspace`-success path: the completion-token check rejects terminal events with a wrong/empty token (no row created), the happy path (correct token, claimed via `claim_next`) materialises the lean row with the payload TTL + registered-provider id + a token-free `spec`, and replay is idempotent. `test_workspace_query_ops.py` covers `get_workspace_claim_state` projection (no `current_holder_run_id` field).

`app/core/workspace/test/test_dispatch_via_workspace_service.py` (service tier) covers `dispatch_via_workspace`: returns `command_id`, `claim_workspace=False` leaves the row unclaimed, `claim_workspace=True` atomically sets `current_command_id`, `WorkspaceNotFoundError` on missing row, `WorkspaceClaimFailed` on busy row, `WorkspaceClaimFailed` on inactive (expired) row. `app/core/workspace/test/test_dispatch_discipline_semgrep_canary.py` (unit, no DB) verifies `dispatch_helper_discipline.yaml` fires on direct `enqueue_command` calls in a domain module's `commands/` files and passes clean Layer-2 callers.

Cross-module tests that need a workspace row without the full provision flow use `seed_workspace` from `app.testing.e2e_setup`.
