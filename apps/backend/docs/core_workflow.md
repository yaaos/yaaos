# core/workflow

> Workflow engine — typed workflows, three command families (AgentDispatch / Local / HITL), async event-driven execution.

## Scope

- **Owns:** `Workflow`, `StepRef`, `WorkflowCommand` types, engine state machine, three taskiq task bodies (`start_step`, `handle_agent_event`, `route_workflow`), `workflow_executions` + `pending_human_decisions` tables.
- **Does not own:** business logic (callers register typed `Workflow` + `WorkflowCommand` impls); workspace provisioning ([`core/workspace`](core_workspace.md)); task durability ([`core/tasks`](core_tasks.md)).
- **Boundary:** callers enqueue work by calling `engine.start()`; the engine routes via taskiq; terminal AgentEvents arrive from [`core/agent_gateway`](core_agent_gateway.md) via `handle_agent_event`.

## Why / invariants

- **Three tasks, not two** — Workspace commands can issue long-running AgentCommands. `start_step` exits after dispatch; workers stay free during the wait; `handle_agent_event` fires when the terminal event arrives.
- **Recovery fires at most once per step instance** — repeated `auth_expired` after recovery has run falls through to Tier-2 retry then Tier-3 transitions. Prevents infinite auth-refresh loops.
- **Finalizer step fires exactly once on terminal-fail, then execution records `failed`.** When `Workflow.finalizer_step_id` is set, the engine dispatches that step before recording `failed`. When the finalizer step completes (even with `transitions={"success": COMPLETE_WORKFLOW}`), the engine checks `finalizer_fired` (typed column) and redirects to the pending failure — never to `DONE`. The original failing step id and failure reason are stashed in `pending_failure_step_id` / `pending_failure_reason` typed columns so they survive the round-trip through the finalizer. On the success path the finalizer step runs as the normal terminal step and transitions normally.
- **`workflow_executions.failure_reason`** — nullable short label written on terminal-fail (e.g. `provision_failed`, `agent_failure`). Populated from the `__failure_reason__` output key first, then the `outcome_label`. Exposed in `WorkflowExecutionSummary.failure_reason`.
- **`workflow.failed` audit row** — written to `core/audit_log` (via `audit(…, org_id=…, session=s)`) on every terminal-fail with payload `{workflow_execution_id, ticket_id, failed_step_id, failure_reason}`. The `org_id` comes from the `org_id` contextvar (set by `OrgContextMiddleware` in task bodies); falls back to `workflow_input.org_id` (the `workflow_input` typed column) when no contextvar is set (e.g. test bodies that bypass middleware), last resort `UUID(int=0)` with a warning log.
- **Workspace commands always dispatch to `awaiting_agent`** — the engine parks the execution and assigns `pending_agent_command_id`; the terminal AgentEvent resumes routing. Provider resolution errors surface in the workspace module's dispatch, not the engine. There is no inline/in-memory dispatch path.
- **`AgentDispatchCommand` is the agent-dispatch seam** — subclasses inherit `@final dispatch(inputs, ctx, *, session) -> UUID`; `start_step` identifies them via `isinstance(cmd, AgentDispatchCommand)`, calls `dispatch`, parks the execution in `awaiting_agent`, and stores the returned `agent_commands.id` as `pending_agent_command_id`. Local + HITL commands are never passed to `dispatch`. Command-to-workflow correlation is stamped via `agent_commands.workflow_execution_id` at enqueue time — `record_agent_event` resolves the workflow from that column with no workspace-row dependency.
- **Typed lambda inputs** — each `StepRef` carries an optional `inputs_factory: Callable[[], BaseModel]` that is called just before the step runs. Inside the lambda, `step_ref.outputs` reads the prior step's typed `Outputs` from `_step_outputs_var` (a ContextVar populated by the engine); `workflow_input_ref.outputs` reads the typed workflow-input snapshot. The public helper `get_step_output(step_id)` returns the outputs or `None` when the step hasn't run yet — used by finalizer lambdas that guard against a missing upstream step. `register_workflow` validates all lambdas at registration time by evaluating them with mock outputs; an `AttributeError` on a non-existent field raises `WorkflowValidationError` immediately.
- **Cross-module callers use read projections**, not raw SQLAlchemy rows. See `WorkflowExecutionSummary`, `HitlHistoryEntry`, `WorkflowRunView`/`WorkflowStepSummary`, and the `list_*` / `get_*` ops in `__init__.py`.
- **`workflow_state_changed` SSE on every transition** — every terminal-state write goes through `_enter_terminal_state(s, wfx, new_state)`, which sets `wfx.state`, calls `_publish_state_changed(s, wfx)` (emits `GeneralEventKind.WORKFLOW_STATE_CHANGED` after commit), and then awaits all registered terminal hooks. All terminal transitions funnel through this single helper; adding a bare `wfx.state = …` outside it is a regression — covered by `test/test_state_changed_sse_service.py`.
- **Per-workflow `on_start` callback** — `Workflow.on_start: WorkflowStartCallback | None`. When set, the engine awaits it inside the bootstrap-commit transaction after `wfx.state = RUNNING` and before `_enqueue_start_step`. Called with keyword-only primitives: `workflow_execution_id: UUID, workflow_name: str, ticket_id: UUID, org_id: UUID, session: AsyncSession`. Must never commit (engine commits after the callback). A raising callback rolls back the entire bootstrap write and records an `exception` event on the `workflow.callback.on_start` span with `StatusCode.ERROR`.
- **Per-workflow `on_terminal` callback** — `Workflow.on_terminal: WorkflowTerminalCallback | None`. When set, the engine awaits it inside `_enter_terminal_state` on every `done / failed / cancelled` transition. Called with keyword-only primitives: `workflow_execution_id: UUID, workflow_name: str, ticket_id: UUID, org_id: UUID, terminal_state: WorkflowState, failure_reason: str | None, session: AsyncSession`. Same commit/raise rules as `on_start`; exception is recorded on `workflow.callback.on_terminal` span with `StatusCode.ERROR` before propagating.
- **Per-step timing lives in `step_state[step_id]`.** `_stamp_step_started` writes ISO-8601 UTC `started_at` when a step is dispatched (after `wfx.current_step_id = step_id` in `_start_step_impl`); `_persist_outputs` writes `completed_at` (preserving `started_at`). These are step-output keys — not engine-internal keys. `WorkflowStepSummary` exposes both as `datetime | None`. Projection: `list_run_views_for_ticket(ticket_id, *, session) -> list[WorkflowRunView]` merges the workflow definition (pending steps), `current_step_id` (the running step), and `step_state[step_id].outcome_label` (`success → done`, `_skipped → skipped`, any other label → `failed`). Pending wins when the execution is in a terminal non-`running`/`awaiting_*` state.

## Gotchas

- `register_workflow` auto-instantiates every command class in `steps` and `recovery_commands` via `cmd_class()` (zero-arg constructor). Constructor args are not supported — use class-level attributes for test-injectable config.
- Duplicate `recovers_failure_label` in `recovery_commands` raises `WorkflowError` at registration time (not at dispatch time).
- `TERMINAL_STATES = {done, failed, cancelled}` — check before enqueuing further work.
- `unregister_workflow` removes a workflow from the process-singleton engine.
- Test isolation uses `scoped_engine` / `scoped_workflow` from [`app/testing/workflow_harness`](../app/testing/workflow_harness.py); those names are no longer on `core/workflow`'s public surface. See [patterns.md § `scoped_*` context managers](patterns.md).

## Vocabulary

- **WorkflowCommand** — base `@runtime_checkable Protocol`; one registered impl per `kind`. Carries `Inputs: type[BaseModel]`, `Outputs: type[BaseModel]`. Engine discriminates between the three command families via `isinstance` — see below.
- **AgentDispatchCommand** — ABC; subclass to get `@final dispatch(inputs, ctx, *, session) -> UUID`. Two concrete branches: `WorkspaceOpCommand` (calls `build_command` to form an agent command) and `CodingAgentCommand` (calls `build_invocation` + dispatches via `core/coding_agent`). Engine path: `isinstance(cmd, AgentDispatchCommand)` → dispatch → `awaiting_agent`.
- **HITLCommand** — ABC; subclass and implement `execute(inputs, ctx) -> Outcome` (no session). Engine path: `isinstance(cmd, HITLCommand)` → execute → must return `hitl_pending` → `awaiting_human`.
- **LocalCommand** — plain structural Protocol; implement `execute(inputs, ctx, *, session) -> Outcome`. Engine path: implicit fallback (not AgentDispatch, not HITL) → execute inline → `route_workflow`.
- **StepRef** — one node in a `Workflow`. Created via `step(CommandClass, inputs=lambda: …)`. `step_id` = `CommandClass.kind`. Carries an optional `inputs_factory` and a `RetryPolicy`. `step_ref.outputs` is a property that reads the ContextVar during lambda evaluation.
- **WorkflowInputRef** — created via `workflow_input(SnapshotClass)`. A synthetic "step 0" whose `.outputs` returns the typed snapshot passed to `engine.start(workflow_input=…)`. Passed as `Workflow(workflow_input=ticket)`.
- **CommandContext** — payload a command sees: execution id, ticket id, step id, attempt counter, optional traceparent.
- **Outcome** — tagged by `OutcomeKind` (`success | failure | hitl_pending`). Carries `outputs` (typed `BaseModel`) and optional failure/hitl fields. `Outcome.failure(reason=…)` sets `failure_reason` on the Outcome; `route_workflow` reads it for `workflow_executions.failure_reason`.
- **`Empty`** — zero-field frozen `BaseModel`; default `Inputs` / `Outputs` type for commands that need no data.
- **`Workflow.finalizer`** — optional `StepRef`; when set the engine routes to that step on terminal-fail (before recording `failed`). One-shot per execution. Already-fired state lives in the `finalizer_fired` typed column.
- **`Workflow.recovery_commands`** — `tuple[type, ...]`; command classes that handle failure-label recovery. Each must declare `recovers_failure_label: ClassVar[str]`. Engine builds a per-workflow map at registration time and auto-registers each class as a dispatchable command.
- **`Workflow.on_start`** — `WorkflowStartCallback | None`; async callable fired inside the bootstrap-commit transaction. Signature (keyword-only): `workflow_execution_id, workflow_name, ticket_id, org_id, session`.
- **`Workflow.on_terminal`** — `WorkflowTerminalCallback | None`; async callable fired inside every terminal-state write. Signature (keyword-only): `workflow_execution_id, workflow_name, ticket_id, org_id, terminal_state, failure_reason, session`.
- **Recovery-policy insertion (Tier-1)** — engine reads `Workflow.recovery_commands: tuple[type, ...]` at `register_workflow` time to build a per-workflow `failure_label → command_class` map. Each recovery command class must declare a `recovers_failure_label: ClassVar[str]` and a `kind` attribute. Duplicate labels within one workflow raise `WorkflowError`. When a step fails with a matching label, the engine inserts a synthetic recovery step with id `_recover_{original_step_id}`, resets the attempt counter, and dispatches the recovery command. Recovery fires at most once per step instance — the engine tracks fired recoveries in `recovered_steps JSONB` column. On success the recovery step returns control to the original step; on failure the workflow fails immediately (no infinite loop). Recovery commands are auto-registered in the engine's command registry at `register_workflow` time — no separate `register_command` call needed.

## Observability — span inventory

All spans emitted by `core/workflow`. Every span carries `org_id` + `yaaos.workflow_id` via the `YaaosDimensionsSpanProcessor` after those values are set on the context.

| Span name | Parent | Where | Notable attributes |
|---|---|---|---|
| `workflow.run.<workflow_name>` | caller's `traceparent` (e.g. the intake request span) | `service.py` `WorkflowEngine.start` | `workflow.name`, `workflow.execution_id`, `workflow.version`; closes when `engine.start()` returns |
| `workflow.command.<Kind>` | `workflow.run.<name>` — opened via `with_remote_parent_span(wfx.otel_trace_context)` so the span is a direct child of the run span regardless of which taskiq worker dequeued the task | `service.py` `_start_step_impl` (AgentDispatch) and `_safe_execute` (Local/HITL) — same name for all families | `command.kind`, `command.category` (`"agent_dispatch"` / `"local"` / `"hitl"`), `workflow.step_id`, `workflow.attempt`; `StatusCode.ERROR` on raise or `Outcome.failure`; for AgentDispatch commands the span closes before `dispatch()` returns so `agent_command.dispatch.<Kind>` is a child of it |
| `workflow.handle_agent_event` | upstream `traceparent` arg (agent's terminal-event request span) | `service.py` `handle_agent_event` task body | `workflow.outcome_label` |
| `workflow.callback.on_start` | `workflow.run.<name>` (via `wfx.otel_trace_context`) | `service.py` `_route_workflow_impl` bootstrap branch | `StatusCode.ERROR` + exception event when callback raises |
| `workflow.callback.on_terminal` | `workflow.run.<name>` (via `wfx.otel_trace_context`) | `service.py` `_enter_terminal_state` | `StatusCode.ERROR` + exception event when callback raises |

`start_step` and `route_workflow` hops are visible as taskiq's auto-emitted `task:workflow.start_step` and `task:workflow.route_workflow` spans. `TaskSpanMiddleware` uses the `metadata.traceparent` stamped by `enqueue` (which reads `wfx.otel_trace_context`) to parent these spans under `workflow.run.<name>` — so `task:workflow.*` spans land in the workflow trace, not in orphan per-task traces. No redundant custom span is opened for either hop. `completed_step_id` and `outcome_label` are stamped as attributes on the `task:workflow.route_workflow` span via `trace.get_current_span()`.

**`otel_trace_context` semantics:** `workflow_executions.otel_trace_context` stores the `workflow.run.<name>` span's own traceparent — not the caller's upstream traceparent. Task bodies use this value when enqueueing downstream tasks so all `start_step` invocations nest under the run span trace, not under the intake request or the agent's HTTP POST. `enqueue` auto-stamps it into `TaskMetadata.traceparent`; `TaskSpanMiddleware.pre_execute` extracts it and uses it as the parent context for the `task:<name>` span — see [`core_tasks.md § Traceparent propagation`](core_tasks.md).

## Trace propagation

`workflow_executions.otel_trace_context` is the **single source of truth** for the `workflow.run.<name>` span's traceparent. Every engine task body that enqueues a downstream task reads it off the loaded `wfx` row:

- **`_start_step_impl` Local branch** — enqueues `route_workflow` with `traceparent=wfx.otel_trace_context`.
- **`_handle_agent_event_impl`** — enqueues `route_workflow` with `traceparent=wfx.otel_trace_context`. Critical site: the caller's traceparent is the agent's HTTP POST request span (a different trace); ignoring it keeps all subsequent command spans nested under `workflow.run.*`.
- **`_enqueue_start_step`** — enqueues `start_step` with `traceparent=wfx.otel_trace_context` on every call path (entry step, retry, recovery step, finalizer, static transition).
- **`resume_hitl`** — follows the same pattern; serves as the canonical reference.

The `traceparent` argument on taskiq task signatures is preserved for backwards-compatibility with in-flight rows but task bodies overwrite it from `wfx.otel_trace_context` after loading. The result: all `workflow.command.<Kind>` spans in a `pr_review_v1` run belong to the `workflow.run.pr_review_v1` trace in Dash0, regardless of which process or HTTP request triggered each enqueue.

## Data owned

- `workflow_executions` — indexes on `state`, `pending_agent_command_id`, `ticket_id`. `failure_reason TEXT` (nullable). Six typed engine-state columns added in migration b3c4d5e6f7a8 (replacing the former JSONB magic-key pattern): `finalizer_fired BOOLEAN NOT NULL DEFAULT FALSE`, `step_attempts JSONB NOT NULL DEFAULT '{}'` (maps `step_id → attempt_number`), `recovered_steps JSONB NOT NULL DEFAULT '{}'` (maps `step_id → failure_label` that triggered recovery), `pending_failure_step_id TEXT`, `pending_failure_reason TEXT` (stash for the original failing context during the finalizer round-trip), `workflow_input JSONB` (typed workflow-input snapshot stored as `model.model_dump()`, passed via `engine.start(workflow_input=…)`). `step_state` JSONB now holds only step-output buckets keyed by `step_id`. Migration 015 (table).
- `pending_human_decisions` — index on `(workflow_execution_id, resolved_at)`. Migration 015.

## How it's tested

`test/test_types.py` — typed data validation (workflows, steps, retry policy, outcomes, terminal transitions).
`test/test_engine.py` — register validation (unknown entry step, dangling transitions, double-register), version-selection, `start()` writes `pending` row + enqueues `route_workflow`, `start()` fails loud on unregistered command. Service tests: `test_engine_state_persists_to_columns_service` (typed columns correct after retry run), `test_migration_backfill_from_step_state_service` (backfill+strip SQL from migration b3c4d5e6f7a8).
`test/test_per_workflow_callbacks_service.py` — `on_start` fires once with expected primitive kwargs; recovery command declared in `recovery_commands` interposes on matching failure label; duplicate recovery label raises `WorkflowError`; raising `on_start` records exception + ERROR on `workflow.callback.on_start` span (error absorbed by drain_once).
`test/test_workspace_dispatch_service.py` — Workspace-branch parks on the dispatched `command_id`, terminal event resumes via `agent_commands.workflow_execution_id`.
`app/core/workspace/test/test_lean_lifecycle_service.py` — finalizer fires exactly once on terminal-fail; success path no refire; `failure_reason` + `workflow.failed` audit row written; original failure context survives the finalizer round-trip.
`test/test_state_machine.py::test_finalizer_runs_then_workflow_records_failed` — finalizer with `"success": COMPLETE_WORKFLOW` in its own transitions still ends in `FAILED`, not `DONE` (production-shape coverage).
`test/test_run_views_service.py` — `list_run_views_for_ticket` projection: state derivation across done/running/pending/failed/skipped branches, terminal-execution-no-outcome resolves to pending (not running), oldest-first ordering, unknown workflow yields empty step tuple.
`test/test_state_changed_sse_service.py` — every transition emits `workflow_state_changed`; the failure path also emits `state=failed`. Catches the most common regression — adding a new state assignment without wiring the publish.
`test/test_workflow_run_root_span.py` — `engine.start()` emits a finished `workflow.run.<name>` span parented to the caller's traceparent, with correct attributes; `otel_trace_context` stores the run span's own traceparent (not the caller's).
`test/test_workspace_command_span.py` — AgentDispatch branch of `_start_step_impl` emits `workflow.command.<Kind>`; `CommandContext.traceparent` passed to `dispatch()` is the command span's own traceparent; dispatch raise → span ERROR + exception event; Local path carries `command.category="local"` attribute.
`test/test_trace_hierarchy_service.py` — every `workflow.command.<Kind>` span is a direct child of `workflow.run.<name>` (same `trace_id` and `parent_span_id == run_span.span_id`) even when `handle_agent_event` is triggered with a foreign traceparent (the agent's HTTP POST context); regression pin for both Local and Workspace branches and for the outbox-to-task-span traceparent pipe (outbox `metadata.traceparent` encodes the workflow trace_id).
`test/test_step_id_attempt_on_command_span.py` — `workflow.command.<Kind>` span carries `workflow.step_id` and `workflow.attempt` on both Local and Workspace branches.
