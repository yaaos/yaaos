# M01 — Modularity (planned)

> Planned modularity model and import rules for both backend (Python) and frontend (TypeScript).
> Not M01-specific: these rules apply to every milestone. Promoted to `docs/modularity.md` when M01 ships.

## Shared rules

These apply to both backend and frontend, with one structural difference: **backend has `core/` + `domain/` + `plugins/`; frontend has `core/` + `domain/` + `shared/`.** Plugins are a backend-only concept (vendor SDKs live there); the frontend uses `shared/` for cross-module reusable primitives. See the stack-specific sections below for details.

- **Three top-level groupings per stack.**
  - `core/` — technical / infrastructure modules. **No business logic.** (Both stacks.)
    - *Nuance:* core modules MAY define domain-aware data types (e.g., a value object whose values reference yaaof concepts like `agent`). What they cannot do is contain business *behavior* — decisions, workflows, rules that change outcomes. **Types-yes, behavior-no.** This is what makes the foundational `core/primitives` module possible.
  - `domain/` — business logic; defines plugin interfaces on the backend, feature modules on the frontend. (Both stacks.)
  - `plugins/` — backend only. Implementations of interfaces defined in `domain/`. The only place vendor-specific code (GitHub, Anthropic, etc.) is allowed to live.
  - `shared/` — frontend only. Reusable UI primitives, hooks, utilities, types. Flat bucket; any module may import from it.
- **Layering on backend:** `core` < `domain` < `plugins`. Each layer may depend on lower layers; never higher.
  - `core` cannot depend on `domain` or `plugins`.
  - `domain` cannot depend on `plugins`. This is what keeps domain vendor-neutral.
  - `plugins` may depend on `core` and `domain`.
- **Layering on frontend:** `core` and `domain` may import from `shared`. `domain/foo` cannot import from `domain/bar`. `core` cannot import from `domain`. Same spirit as the backend layering.
- **Modules talk only through interfaces.** A module's interface is its `__init__.py`'s `__all__` (Python) or its `index.ts`'s named exports (TypeScript). Importing internals across module boundaries is forbidden.
- **Absolute imports only.** No relative imports across module boundaries.
- **No self-imports from `__init__.py` / `index.ts`** within the same module. Use direct internal paths.
- **Tests live inside the module** (`<module>/test/`).
- **When `core` needs `domain` capability, use a registry.** `core` defines a registry; `domain` modules register themselves at import time. Keeps `core` domain-neutral while still routing work to `domain` modules.
- **Plugin registration (backend only).** Each pluggable concern in `domain/` (e.g., `domain/vcs`, `domain/llm`, `domain/executor`) defines a Protocol and a registry. Plugins in `plugins/` register themselves at import time. The bootstrap is responsible for importing the plugin packages that should be active.

## Backend specifics

- Subdirectories of `apps/backend/app/core/`, `apps/backend/app/domain/`, and `apps/backend/app/plugins/` are each a module.
- **Each module owns its own DB tables.** Single Postgres database; ownership is by convention (and enforced by where SQLAlchemy models live). Plugin-specific tables are owned by the plugin (e.g., `plugins/github` owns GitHub App install state).
- **Module documentation lives in `docs/<module>.md`** at the repo root, not inside the module directory.
- **Tach configuration** (`apps/backend/tach.toml`) is generated by `bin/sync_modules` — never hand-edit.

## Frontend specifics

- Three top-level groupings under `apps/web/src/`:
  - `core/` — routing, layout, auth, global state, observability, websocket/SSE plumbing.
  - `domain/<feature>/` — pages, components, stores, hooks, types for one feature.
  - `shared/` — reusable UI primitives, cross-domain hooks, utilities, icons, types, assets.
- **Cross-domain imports are forbidden.** `domain/foo/` cannot import from `domain/bar/`. Move the shared piece to `shared/` or extract a new module.
- **Biome custom rule** (or ESLint fallback) enforces cross-domain and layering rules in CI.

## Tooling: `bin/sync_modules`

Single command runs the complete modularity workflow:

1. **Discover modules** — scan `core/`, `domain/`, and `plugins/` subdirectories on both sides.
2. **Sync `tach.toml`** — write module entries and interface exports based on `__init__.py` `__all__`.
3. **Check internal imports** — fail on relative imports across boundaries or self-imports from a module's `__init__.py`.
4. **Check layering** — fail if a `core/` module imports from `domain/` or `plugins/`, or if a `domain/` module imports from `plugins/`.
5. **Run `tach check`** — fail on undeclared dependencies or imports of non-interface symbols.
6. **Frontend equivalent** — run Biome's custom rule for cross-domain imports and layering.

CI runs the same workflow on every PR.

## Adding a new module

1. Create the directory under `core/`, `domain/`, or `plugins/` (and `shared/` on FE if applicable).
2. Add `__init__.py` (Python) or `index.ts` (TS) with the public interface declared.
3. For a new plugin, ensure the bootstrap imports the plugin package so its registration runs.
4. Run `bin/sync_modules`.
5. Add `docs/<module>.md` describing responsibility, public interface, owned data, and how it's tested.

## Decisions

### 2026-05-13 — Modularity model adopted
Core/domain split on backend; core/domain/shared on frontend; interface-only imports enforced by tach (BE) and Biome custom rules (FE); registry pattern for `core` → `domain`.
**Why:** the principle is platform-agnostic and applies cleanly to both sides; same vocabulary on BE and FE cuts cognitive load.

### 2026-05-13 — Full `bin/sync_modules` workflow
Auto-discover modules, generate `tach.toml`, check internal imports, check layering, run `tach check`. Hand-editing `tach.toml` is forbidden.

### 2026-05-13 — Plugins as a third top-level grouping
`core` < `domain` < `plugins`. `domain` defines plugin interfaces (Protocols + registries); `plugins` are the only place vendor-specific code lives (GitHub, Anthropic, etc.).
**Why:** keeps `domain` vendor-neutral so swapping or adding providers (GitLab, OpenAI, …) is a config change, not a refactor.

### 2026-05-14 — Core may define domain-aware data types; business logic stays out
The "no business logic in core" rule is about *behavior*, not *type awareness*. A core module may define a value object whose values reference yaaof concepts (e.g., `Actor` with `ActorKind ∈ github_user | agent | system`). What it cannot do is encode decisions, workflows, or rules.
**Why:** without this nuance, foundational types like `Actor` get orphaned — every candidate home pulls them across a layer boundary. Clarifying the rule lets `core/primitives` exist cleanly.
